# Процесс ведения проектов внедрения (маркировка) в 1С:Документооборот («ДО»)

Извлечено из папки `ДО` (2026-07-10). Источники: `flowchart_DO.html` (блок-схема процесса),
`Работа с проектом.xlsx` (чек-листы приёмки/передачи), `Создание проекта в ДО.docx` (конспект
вебинара 03.10.2025 «Работа с 1С Документооборот для проектного управления»).

**Что это.** ДО = **1С:Документооборот** — система, в которой команда ведёт «запуски»
(проекты внедрения маркировки: установка оборудования, камеры, производственные линии, КИЗ,
Честный Знак). Роли на проекте: **Менеджер/Аккаунт · Внедренец · Монтажник/Инженер** (минимум 3,
известны заранее). Это delivery-сторона, отдельная от пресейл-сделки.

---

## 1. Состояние запуска (жизненный цикл) — ТОЧНО, 2 источника

```
Инициирован  →  Планируется  →  Исполняется  →  Завершён
```

- **Инициирован** — при создании проекта (дефолт).
- **Планируется** — идёт составление плана (режим «Планирование»: двигают сроки, длительности).
- **Исполняется** — активная фаза (режим «Контроль выполнения»: проставляют факт задач).
- **Завершён** — проект закрыт.

⚠️ Это НЕ линейная воронка этапов (как Обследование→ОПЭ). Это **4 крупных статуса всего
проекта**. Детализация работ живёт отдельно — в **плане проекта** (дерево задач), см. §2.

---

## 2. Два уровня, не путать (важно для CRM)

| Уровень | Что это | Аналог в CRM |
|---|---|---|
| **Состояние** (4 шт.) | статус всего запуска | пайплайн `entity_type='project'` — но 4 стадии, не 8 |
| **План проекта** | дерево задач (1.1, 1.1.1…) со сроками, зависимостями, ответственными | доска задач PCT-1 (`project_columns` / `tasks`) |

План: задачи параллельные/последовательные; при сдвиге срока одной — план пересчитывается;
можно планировать от даты окончания. Примеры задач плана: «1.1.1 Открыть проект в
информационных системах», «Отгрузить оборудование клиенту» (с адресом/датой доставки в свойствах).

---

## 3. Атрибуты запуска (реквизиты при создании)

- **Наименование** (ЕДИНОЕ везде: проект = папка запуска = заказ клиента), формат:
  `[номер]. Запуск [дата] [клиент] [товарная группа]` → напр. `01. Запуск 21.11.2025` или
  `01-01.12.25 Forward Велосипеды`.
- **Ссылка на заказ клиента** — из системы Шоурум.
- **Руководитель проекта** — обязателен.
- **Даты**: старт + окончание.
- **Команда** (3 роли, критично, утверждены заранее): Менеджер, Внедренец, Монтажник.
- **Настройки хранения**: папка клиента в ДО.
- Создание чаще всего — **копированием из шаблона** («Шаблоны запусков» → копировать с задачами
  и контрольными точками). ← это ровно паттерн «шаблон проекта» (PCT-1 P3).

---

## 4. Ключевые гейты / вехи (формальные приёмки)

Проект проходит два формальных подтверждения Администратора (чек-листы из xlsx):

**A. Подтверждение корректности документации по запуску** — проверка запуска + папки клиента:
наименование по формату, ссылка на заказ, руководитель, актуальное состояние, команда,
артефакты в задаче 1.1.1, актуальные сроки/ответственные, адрес+дата доставки оборудования;
структура папки клиента, договор в «00. Администрация», папка запуска, оргвстречи+резюме.

**B. Передача на поддержку** (гейт перед завершением) — чек-лист из 11 пунктов, часть проверяет
поддержка: Список ответственных лиц, Пользователи базы 1С, Пользователи производства,
оргвстречи+резюме, согласованный с РП заказчика список открытых вопросов, протоколы
обучения/аттестации по базе знаний, Порядок обновления, настройки камер (в облако),
производственные линии (фото/видео), подключение к базе 1С, чек-лист БИТ.MDT.

→ То есть перед «Завершён» есть **вешка «Передача на поддержку»** с sign-off, а после — фаза
**Поддержка**. Это важные состояния, которых нет в голом «Инициирован→…→Завершён».

---

## 5. Процесс работы (5 блоков вебинара)

1. **Создание проекта** — из шаблона, реквизиты, команда, папка хранения.
2. **Работа с планом** — режимы Планирование / Контроль выполнения; связи задач; автопересчёт.
3. **Выполнение задач** — карточка задачи, артефакты (ссылки на папку клиента, чат в Telegram,
   скриншоты, подтверждение прав админа), проставление факта.
4. **Работа с документами** — папка клиента (структура-шаблон: 00. Администрация, 01. Оргвстречи,
   папка запуска с чек-листами и файлами).
5. **Отчёты и завершение** — «Статус отгрузок» (логистика), передача на поддержку, завершение.

Путь в системе: `Совместная работа → Проекты`. Фильтр проектов по состояниям и по руководителю.

---

## 6. Что это значит для delivery-модуля CRM (честный вывод)

1. **Реальные «этапы» delivery — это 4 состояния, а не 8-стадийная воронка.** Засиженный в
   вашей БД пайплайн `project/erp` «ERP Внедрение» (Kickoff→Обследование→…→Поддержка, 8 стадий)
   и `project/iiot` (Kickoff→Подключение КИЗ→…, 5 стадий) — **не совпадают** с фактическим
   процессом ДО (Инициирован→Планируется→Исполняется→Завершён + гейт «Передача на поддержку»).
   Кто-то их насочинял; перед активацией — пересмотреть под реальность.

2. **Система-запись для delivery уже есть — это 1С:Документооборот.** Команда ведёт запуски там:
   планы, папки клиента, чек-листы, приёмки. Дублировать это в CRM 1:1 — бессмысленно и вредно.
   **Ключевой вопрос к тебе:** CRM-модуль «Проекты» должен (а) **зеркалить** статус запуска из ДО
   (4 состояния + ссылка на проект в ДО, тонкий слой видимости для продажников/руководства), или
   (б) **заменить** ДО (маловероятно — там весь документооборот, чек-листы, аттестации)?
   Мой предварительный совет — (а): CRM держит сделку и лёгкую карточку проекта со статусом +
   ссылкой в ДО, а тяжёлый delivery остаётся в 1С:ДО. Интеграция (статус/ссылка), не копия.

3. **Что переиспользуется из уже сделанного:** шаблоны запусков ДО ≈ ваш PCT-1 P3 (шаблоны
   проектов); план задач ДО ≈ доски PCT-1 (`project_columns`); 3-ролевая команда ≈ ваши
   memberships/ownership. Основа под лёгкий проектный модуль у вас есть.

## 7. Канонический СДР (план запуска) из 1С:ДО — фактические этапы

Источник: отчёт «Контроль выполнения задач проекта» (скриншоты 10.07.2026). Две вариации
плана: полный **Запуск** и укороченный **Эксперимент** (пилот). Это шаблон, копируемый при
создании; лишние задачи гасятся префиксом «НЕ ТРЕБУЕТСЯ!».

**Верхний уровень (4 фазы СДР):**
1. **Подготовка к запуску** — 1.1 Подготовительный этап · 1.2 Установка расширения БИТ.MDT
   (ЭП с МЧД для ЧЗ, серверная подпись, ввод в оборот) · 1.3 Подготовка оборудования (сценарии
   тестирования, аттестация пользователей 1.3.10, отгрузка оборудования 1.3.11-12) ·
   1.4 «1 неделя до запуска» · 1.5 «2 дня до запуска».
2. **Запуск** — 2.1 Запуск · 2.2 Подписать протоколы тестирования/обучения · 2.3 Список камер ·
   2.4 Post Mortem · 2.5 Завершение запуска.
3. **Регулярные мероприятия** (⚠️ идут ПАРАЛЛЕЛЬНО, даты перекрывают всё) — 3.1 Еженедельная
   конференция · 3.2 Консультация и решение задач Заказчика.
4. **Передача на поддержку** — 4.1 Список открытых вопросов · 4.2 Подтверждение администратора ·
   4.3 Подтверждение службы поддержки · 4.4 Встреча · 4.5 Чек-лист передачи на поддержку.

**Эксперимент** = тот же план короче (пилот перед полным запуском). Многие задачи помечены
«НЕ ТРЕБУЕТСЯ ЭКСПЕРИМЕНТ».

**Состояние задач в отчёте:** Завершены / Выполняются с просрочкой / Не начаты (это состояние
ЗАДАЧИ, не проекта).

### Вывод для модели статуса CRM (пересобрано под факт)

⚠️ **Delivery НЕ линеен** (фаза 3 «Регулярные» параллельна, задачи 1.x/2.x перекрываются) —
поэтому линейный стейдж-грид как у сделки НЕ подходит. Правильная модель:

- **4 колонки канбана проектов = Состояние проекта:** Инициирован → Планируется → Исполняется
  → Завершён (грубый портфельный разрез, синк с 1С).
- **Детализация внутри проекта = план-дерево**, как «Контроль выполнения задач» в 1С:
  4 фазы-секции СДР (Подготовка / Запуск / Регулярные / Передача) → задачи → состояние каждой
  (Не начата / Выполняется / Просрочена / Завершена) + прогресс (X/Y, N просрочено).
- Точность даёт **план + прогресс**, а не нарезка на под-этапы. Ложится на PCT-1 доски
  (`project_columns` = фазы СДР, `tasks` = задачи).

### Механизм шаблона (для PCT-1 P3)

Шаблон = супермножество всех задач; создание проекта = копия шаблона + выключение неприменимых
(«НЕ ТРЕБУЕТСЯ!»). Два шаблона: Запуск (полный) и Эксперимент (пилот).
