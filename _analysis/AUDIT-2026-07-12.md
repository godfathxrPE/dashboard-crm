# Комплексный аудит dashboard-crm — 12.07.2026

**Скоуп:** код (архитектура, Supabase-схема/RLS, data-слой, темизация) + визуал (aura как основная, тёмные темы frost/aurora/tidal). Scandi/paper/sand не аудировались по содержанию — только стоимость удаления.
**Метод:** 4 параллельных код-агента по актуальному коду + живой осмотр localhost:3000 и прода (Netlify) на 17 экранах. Прошлые аудиты (14.06) учтены: закрытое не включено, незакрытое помечено.
**Формат:** проблема → где в коде/схеме → почему важно.

# ЧАСТЬ 1. КОД

## P1 — безопасность и потеря данных

### 1.1. Живой refresh-token владельца прода в корне репо
- **Где:** `token.json` (корень) — ключи `at` (JWT, истёк) и `rt` (refresh-token сессии mzk.oleg@gmail.com, инстанс uoiavcabxgdjugzryrmj).
- **Почему важно:** refresh-token не истекает сам; любая копия папки (бэкап, синк, передача на аудит — как сейчас) = живая сессия владельца орги со всеми правами RLS. Файл в .gitignore, но физически разъезжается.
- **Действие:** Supabase → Authentication → Sessions → revoke; удалить файл. Сделать до всего остального.

### 1.2. Цепочка миграций не воспроизводима и содержит невычищенные RLS-дыры
- **Где (составные части):**
  - `supabase/migrations/005_calls_meetings.sql:75` — `CREATE POLICY "attendees_all" ... FOR ALL USING (true)` на `meeting_attendees`; ни одна миграция её не дропает (023 дропает только прод-имя `attendees_own`).
  - `009_delete_triggers.sql:135-140` — `WITH CHECK (true)` на insert в `activity_log`; дропа нет.
  - Объекты «Sprint 1» (`pipelines`, `pipeline_stages`, `projects.stage_id/status/...`, `sync_deal_stage_fields`) не создаются ни одной миграцией, но используются с `016_leads.sql:117` — реплей падает уже на 016.
  - `001_profiles.sql:32-38` — политики, о которые споткнётся `024:313` (DROP COLUMN role).
  - Функции из 001/007/009/011/016 — без `SET search_path` и revoke от anon (прод-фиксы 14.06 не бэкпортированы).
- **Почему важно:** фиксы июня применялись на прод вручную через MCP и в репо не вернулись. Репозиторий перестал быть источником истины: staging/DR/локальная БД из миграций либо не собирается, либо собирается с открытой на весь инстанс таблицей участников встреч (PII, 152-ФЗ). Все RLS-гарантии сейчас существуют только в живой БД.
- **Действие:** одна ревизионная миграция-«снимок» (dump прод-схемы как новая базовая точка) либо бэкпорт всех прод-фиксов; далее правило «фикс = миграция в репо, MCP только применяет».

### 1.3. Все 8 модалок-форм теряют введённые данные без предупреждения
- **Где:** `CallModal.tsx:156`, `CompanyModal.tsx:104`, `ContactModal.tsx:107`, `MeetingModal.tsx:130`, `LeadModal.tsx:94`, `LeadConversionModal.tsx:139`, `ProjectModal.tsx:262`, `TaskModal.tsx:116` — оверлей `onClick={onClose}` без проверки `formState.isDirty` (grep по всем — 0 вхождений).
- **Почему важно:** менеджер 10 минут заполняет итоги звонка, промахивается мимо кнопки — текст исчезает молча. Для CRM это самый дорогой тип бага: подрыв доверия к вводу данных.

### 1.4. Ошибки сохранения молчат: console.error + тихий откат optimistic-апдейтов
- **Где:** `onSubmit` всех модалок — `catch (err) { console.error }` (`CallModal.tsx:147-149`, `CompanyModal.tsx:86-88`, `ContactModal.tsx:99-101`, `ProjectModal.tsx:253-255`, `MeetingModal.tsx:122-124`, `LeadModal.tsx:81-83`, `TaskModal.tsx:96-110`); из 88 `.mutate(` в components только ~16 с `onError`; optimistic-откаты без уведомления: `use-companies.ts:135,152,170`, `use-tasks.ts:139,213,262` и др. Toast-системы в проекте нет.
- **Почему важно:** при RLS-отказе/обрыве сети запись «создалась» (optimistic), через секунду тихо исчезла. В связке с 1.5 (протухшая сессия) — основной сценарий системной тихой потери ввода.
- **Действие:** один toast-провайдер (sonner) + `onError` в mutation defaults QueryClient — закрывает весь класс проблем одной точкой.

### 1.5. Realtime-синхронизация ломается насовсем при первом же уходе со страницы
- **Где:** `src/lib/hooks/use-realtime.ts:20-33` — канал с именем `realtime-${table}` разделяется всеми подписчиками (realtime-js дедуплицирует по topic), unmount любого вызывает `removeChannel` для всех; повторный subscribe копит дубли bindings → после reconnect канал уходит в errored. Одновременные подписчики одной таблицы: `use-tasks.ts:20,68` + постоянные в layout (`EventReminder.tsx:24-26`, `ActivityDrawer.tsx:120-122`, `CommandPalette.tsx:93-98`).
- **Почему важно:** ушёл со страницы /tasks — realtime для tasks умер до конца сессии у всех layout-компонентов; после сна ноутбука — совсем. Второй пользователь двигает сделку, первый смотрит на устаревшую доску. Параметр `_queryKey` игнорируется — мёртвый контракт.
- **Действие:** уникальные имена каналов (`realtime-${table}-${useId()}`) или один менеджер подписок на приложение с refcount.

### 1.6. Excel-импорт падает посередине и врёт об успехе
- **Где:** `ExcelImport.tsx:167-207` — `const { data: newCo } = await insert...` без проверки `error`, затем `newCo!.id` (TypeError при отказе); ошибки update/upsert игнорируются; `executeImport` без try/catch (`:324`).
- **Почему важно:** импорт 200 строк падает на 57-й — UI застревает на «importing», в БД частично залитый набор, «успешные» строки могли не сохраниться. Тестовые артефакты в проде (`<Клиент 129>`, `NN/qqqqq` на экране Компаний) — похоже, следы таких прогонов.

## P2 — надёжность и целостность

### 2.1. Протухшая сессия не обрабатывается
- **Где:** `use-auth.ts:19-27` — единственный `onAuthStateChange`, только `setUser`; `QueryProvider.tsx:8-19` — нет глобального onError; `qc.clear()` нигде.
- **Почему важно:** вкладка с вечера утром показывает кеш или «пустую CRM» под anon-RLS; мутации молча падают (см. 1.4). Редирект на /login — только при навигации.

### 2.2. Drag-and-drop: шторм из N параллельных мутаций на один drop
- **Где:** `KanbanBoard.tsx:182-192` — `arrayMove().forEach((t,i) => updateTask.mutate(...))`; каждая инвалидирует `['tasks']` (`use-tasks.ts:262-270`).
- **Почему важно:** до 15-30 PATCH на перенос; откат одной мутации затирает соседние успешные; порядок карточек «прыгает» после рефетча промежуточного состояния. Нужен bulk-RPC/одна мутация с массивом.

### 2.3. INSERT-политики ai-hub не фиксируют org_id
- **Где:** `030_ai_hub.sql:83-90` (`transcripts_insert`), `:108-114` (`ai_runs_insert`) — WITH CHECK без org-проверки; триггер `trg_set_org_id` (`022:55-63`) заполняет только NULL, явное чужое значение проходит.
- **Почему важно:** единственные tenant-таблицы без org-границы на INSERT: можно вставить строку с чужим org_id — «потерянные» данные и фантомные ai_runs в чужой статистике токенов.

### 2.4. Приглашения: членство выдаётся по email без проверки токена
- **Где:** `026_notifications_invitations.sql:212-247` — `apply_pending_invites` матчит по `lower(email)`, поле `token` (`:24`) в коде не участвует.
- **Почему важно:** кто первым зарегистрирует аккаунт на приглашённый адрес (при выключенной верификации почты — кто угодно), тот получает membership с ролью admin/manager в чужой орге.

### 2.5. Legacy `stage`: половина UI читает документированно смещённое зеркало
- **Где:** маппер `stage-mapping.ts:18-33` (смещение признано в комментарии `:8-16`); читатели: `ProjectCard.tsx:97-107`, `ProjectsTable.tsx:36,72-74`, `ProjectDetail.tsx:334-336`, `FunnelWidget.tsx:15`, `WeeklyReview.tsx:36-37`; аналитика при этом уже на `stage_id` (`Charts.tsx:186`). Плюс триггер `log_stage_change` реагирует на зеркало, а не на факт (`024` vs `032/035`, где `stage := NULL` для internal/delivery).
- **Почему важно:** для IIoT-стадий после точки расхождения карточка/таблица/воронка показывают неверную стадию; таймлайн смен стадий в activities неполный. «Путь B» из SPRINT-stage-deprecation так и не выполнен — известно с июня.

### 2.6. Удаление проекта с историей активностей падает
- **Где:** `006_activities.sql:12` — `project_id REFERENCES projects(id)` без ON DELETE (дефолт RESTRICT), при этом у `activities` нет DELETE-политики (`006:38-45`, подтверждено в `023:225-238`) — почистить клиентом нельзя. Также без ON DELETE: `scheduled_calls.*`, `kpi_entries.profile_id` (`007:8,32-41`).
- **Почему важно:** DELETE FROM projects → 23503, тупик, разрешимый только service-ролью. Остальной граф системно на SET NULL/CASCADE — эти пропущены.

### 2.7. Все данные тянутся клиентом целыми таблицами на каждой странице
- **Где:** layout монтирует `EventReminder`/`ActivityDrawer`/`CommandPalette` (`layout.tsx:90-95`), те тянут `useCalls`/`useMeetings`/`useTasks`/`useProjects`; `select('*')` без `.limit()`: `use-calls.ts:56-63`, `use-contacts.ts:47-60`, `use-companies.ts:43-51`, `use-meetings.ts:53-59`. DataTable пагинирует только рендер. RSC Next 15 не используются — серверные страницы лишь auth-обёртки.
- **Почему важно:** вся история звонков с join'ами скачивается при открытии страницы настроек. На 5-10 тыс. записей — секунды и мегабайты на каждый визит; деградация ползучая.

### 2.8. Hydration mismatch для всех, кроме владельца
- **Где:** `(dashboard)/layout.tsx:55-83` — выбор shell (`ScandiSidebar` vs `Sidebar`, наличие Header, ml-классы) по persisted-теме из zustand в рендере; SSR рендерит дефолт t-scandi. Inline-script в `app/layout.tsx:72` чинит только CSS-класс, не структуру дерева.
- **Почему важно:** у пользователя с тёмной темой каждая загрузка = hydration error + полный ре-рендер с миганием «scandi-shell → frost». У тебя (aura → текстовый nav, та же ветка что scandi) баг маскируется.

### 2.9. Инвалидация кеша дырявая
- **Где:** `['dashboard-stats']` не инвалидируется никем и глотает ошибки счётчиков (`dashboard-content.tsx:27-55` — не исправлено с 14.06); `['timeline',...]` инвалидируется единственным местом (`ProjectDetail.tsx:186`), мутации calls/meetings/tasks его не трогают.
- **Почему важно:** создал сделку — KPI не изменились; залогировал звонок из хаба контакта — в ленте на том же экране его нет. Выглядит как «не сохранилось», провоцирует повторный ввод.

### 2.10. `notif_update` без WITH CHECK
- **Где:** `026:85-88` — UPDATE-политика без WITH CHECK: получатель может сменить recipient_id/org_id/payload.
- **Почему важно:** ущерб низкий (уведомления), но это дыра в дисциплине WITH CHECK; защита держится на клиентском типе `Update = {read_at}`.

# ЧАСТЬ 2. ВИЗУАЛ

*Разбор по рубрике PAL + dashboard-addendum. Основная тема — aura light; тёмные — frost/aurora/tidal. Прод отстаёт от локала минимум на одну фичу (нет чипа «Просроченные») — аудит по локалу.*

## Первое впечатление

Aura — зрелая работа: непрозрачные карточки на атмосферном фоне (приём Linear/Stripe), дисциплинированная сетка, продуманная пара Onest + Unbounded, у «Сегодня» есть настоящая точка входа (Фокус дня). Это уже не «дашборд из шаблона». Проблемы — не в основе, а в двух местах: сигнальная система цвета размыта декором, и тёмные темы живут как отдельный продукт с другим шеллом и другими багами.

## Что работает

- **Токен-ядро чистое** (MEASURED): семантические `*-text`-варианты дают контраст текста 18.0:1 (основной), ≥4.5 на всех статусных парах aura; в data-зонах ноль утечек сырой Tailwind-палитры.
- **Сканируемость «Сегодня»**: фокус дня → задачи в работе → остывающие контакты. Правильная информационная драматургия для утра менеджера.
- **Spacing-дисциплина**: произвольных padding/gap — 2 шт. на весь проект; сетка держится.
- **tabular-nums** в таблицах и KPI есть (реализация — см. 2П-7).

## P1 — визуальные

### 1П-1. Белый текст на жёлтых заливках нечитаем во всех темах
- **Где:** оба защитных слоя перечисляют accent/green/red/blue/purple, но не yellow: `globals.css:274-278` (aura), `:1873-1887` (dark); жертвы — `PomodoroWidget.tsx:112` (кнопка «Пауза»), `MeetingsList.tsx:195` + `globals.css:1817-1819` (дата-бейдж ближайшей встречи в aura).
- **Значения (MEASURED):** белый на --yellow: aura 3.50:1, tidal 2.56:1, frost 1.65:1, **aurora 1.31:1** — в aurora текст практически исчезает. Норма 4.5:1.
- **Почему важно:** работающий таймер и дата ближайшей встречи — состояния, требующие внимания; именно они наименее читаемы. [ESTABLISHED — WCAG 1.4.3]

### 1П-2. Дефолтная тема — t-scandi, которая идёт под удаление
- **Где:** `theme-store.ts:16`, `ThemeProvider.tsx:11-13` (fallback), `app/layout.tsx:57` (SSR-класс).
- **Почему важно:** блокер плана удаления. Удалишь scandi без миграции дефолта → сломанный первый рендер у всех, у кого localStorage пуст или хранит scandi. Порядок: сначала дефолт → t-aura + миграция persisted-стора, потом удаление.

### 1П-3. В 7 темах из 9 таблицы превращаются в стену призрачных «+ добавить»
- **Где:** `.add-on-hover` скрывается только для scandi и aura (`globals.css:1305-1311`); в frost/aurora/tidal/washi/fuji/paper/sand ghost-CTA видимы во ВСЕХ пустых ячейках (`CompaniesTable.tsx:139,156,173`, `ContactsTable.tsx:124,141`). Подтверждено скриншотом frost /companies.
- **Почему важно:** таблица на 75 строк × 3 пустых колонки = 200+ одинаковых интерактивных надписей; data-ink переворачивается — хром громче данных. Классика Тафти наоборот. [ESTABLISHED — data-ink ratio, Tufte]

### 1П-4. Модалка «Сделка» не помещается во вьюпорт — заголовок и submit за экраном
- **Где:** VISUAL_ESTIMATE, воспроизведено на 1514×784 (реальная высота твоего окна): верх модалки (title, close) обрезан сверху, footer с кнопками — снизу, wheel-scroll внутри тела не прокручивает. `ProjectModal.tsx` — самая длинная форма (585 строк, все поля в один столбец).
- **Почему важно:** на ноутбучных вьюпортах (768–800px) форму создания сделки физически неудобно отправить/закрыть штатно. В связке с код-1.3 (клик мимо = потеря данных) — худший сценарий на самом частом действии CRM.
- **Фикс:** `max-h-[calc(100dvh-4rem)]` + sticky header/footer + `overflow-y-auto` на теле формы.

## P2 — консистентность

### 2П-1. Сигнальная система цвета разбавлена декором
- **Где (VISUAL_ESTIMATE по /deals):** «КОНВЕРСИЯ 50%» — единственный цветной KPI (зелёный) без семантической причины; заголовки 4 стадий воронки — 4 разных хью (серый/фиолетовый/янтарный/синий) при том, что цвет стадии ничего не кодирует; в тёмных темах эти хью ещё и сатурированнее.
- **Почему важно:** глаз учится игнорировать цвет → настоящие сигналы (won/lost/overdue) теряют силу. Правило: в data-зоне цвет = семантика, декор — только в атмосферном слое (орбы). [ESTABLISHED — signal dilution; ср. Linear: один акцент + статусные]

### 2П-2. Цвета aura-бренда живут в 4 несинхронизированных источниках
- **Где:** `globals.css:1582-1651` (CSS rgb), `AuraOrbs.tsx:18-31` (WebGL float, комментарий «синхронно с globals.css» = ручная синхронизация), `Charts.tsx:33-38` (hex), `DashboardHome.tsx:144-147` (hex).
- **Почему важно:** смена акцента раздела = правка 4 файлов в 3 системах счисления; расхождение даст разный цвет CSS-орба и WebGL-орба, лежащих друг на друге.

### 2П-3. /tasks: интерфейсный хром тяжелее данных
- **Где (VISUAL_ESTIMATE):** часы 18:35 — самый крупный и жирный элемент экрана при нулевой информационной ценности; «Готово» повторено 9 раз подряд в списке.
- **Почему важно:** визуальный вес должен следовать за приоритетом информации, а не за размером виджета. Кнопку действия — на hover/фокус строки; часы — в ряд мелких метаданных. [ESTABLISHED — визуальный вес; данные важнее хрома]

### 2П-4. Тёмные темы — другой продукт: второй сайдбар-шелл
- **Где:** aura → `ScandiSidebar` (текст-капс, «Dashboard БИТ.IIOT»), dark-темы → `Sidebar` (иконки, «Torii CRM»); ветвление `layout.tsx:55-83`; это же источник hydration mismatch (код 2.8).
- **Почему важно:** переключение темы меняет навигационную моторику и брендинг — темы должны менять кожу, не скелет. Плюс двойная стоимость поддержки каждой правки нава.

### 2П-5. Артефакты тёмных тем
- **Где (VISUAL_ESTIMATE, скриншоты):** tidal /tasks — угловые скобки `「」` (`--bracket-color`, `globals.css:1161-1260`) висят в пустоте между секциями как рендер-баг; frost «Сегодня» — тёмная полоса-артефакт за заголовком; card-борта в dark: border/surface 1.13–1.67:1 (MEASURED) — карточки держатся только на тенях, на дешёвых матрицах сливаются.
- **Почему важно:** decorative-слой, спроектированный для light, перенесён в dark без ревизии.

### 2П-6. Микротипографика вне шкалы
- **Где (MEASURED):** 273 arbitrary-размера: `text-[10px]`×147, `text-[11px]`×48, `text-[9px]`×13, `text-[8px]`×1; `extend.fontSize` в tailwind.config отсутствует. Де-факто шкала 10/11/12/14 существует как привычка, не как токены.
- **Почему важно:** 8–9px ниже порога читаемости; шкалу нельзя править централизованно. [ESTABLISHED — типографическая шкала, Müller-Brockmann]

### 2П-7. tabular-nums навешан хаком на весь жирный текст
- **Где:** `globals.css:1484` — `.font-bold, .font-semibold { font-variant-numeric: tabular-nums }`.
- **Почему важно:** таблицы покрыты (хорошо), но заголовки получают табличные цифры и теряют плотность набора. Токен должен жить на data-ячейках.

### 2П-8. Выигранная сделка не выглядит выигранной
- **Где (VISUAL_ESTIMATE, deal detail ОМК):** все 12 чевронов стадий серые, галочки микроскопические, прогресс-бар — чёрная полоса 100%, «Выиграна · 100%» — мелкий текст.
- **Почему важно:** статус — главный факт карточки, а прогрессия его не кодирует; чёрный full-width бар читается как ошибка рендера, а не как успех.

### 2П-9. Мелочи консистентности
- Заголовки таблицы компаний в двух регистрах («Компания, Отрасль» vs «ТЕЛЕФОН, EMAIL»).
- Fuji держит второй словарь статусов (`--danger/--success/...` только в fuji) — использование `var(--danger)` вне fuji даёт пустое значение.
- Маппинг статус→цвет размазан по 8 компонентам и уже разошёлся (pill-pending=blue в `globals.css:1977` vs pending=yellow в `CallLog.tsx:89`).
- Пустые чарты рендерят голые оси 0–4 вместо empty state (overview, analytics).
