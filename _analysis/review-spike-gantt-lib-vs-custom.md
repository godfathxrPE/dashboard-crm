## С чем согласен полностью

### 1. Тема — доминирующий аргумент, и он верен для нашего стека

Спайк ставит CSS-переменные / Tailwind-токены на первое место — это совпадает с `theme-system.md` и фактическим v0:

- `GanttTimeline.tsx` уже на `bg-accent`, `bg-red`, `border-border`, `text-text-mute` — без per-theme override.
- В `theme-store.ts` (AUDIT C) **активны 6 тем**: `t-aura`, `t-washi`, `t-fuji`, `t-frost`, `t-aurora`, `t-tidal`; `scandi/paper/sand` — legacy с миграцией на дефолт. Упоминание «6 тем» в спайке **актуальнее**, чем «9 тем» в старом `crm-architect` — для Gantt-решения это правильный фокус.
- Три dark-glass темы (`frost`, `aurora`, `tidal`) с полупрозрачным `--surface` — реальный усложнитель для любой чужой либы; спайк это учитывает.

Вывод по frappe / @wamra / SVAR — справедлив: override чужого DOM/props на каждой теме = скрытый налог, который съедает выигрыш «из коробки».

### 2. Таблица кандидатов — достаточная для решения

| Кандидат | Ревью согласия |
|----------|----------------|
| frappe-gantt | ✅ SVG + свои стили, client wrapper — типичная боль |
| @wamra/gantt-task-react | ✅ props/inline colors, SSR не заявлен — совпадает с pushback из `REVIEW-grok-crm-gap` |
| SVAR React Gantt | ✅ Резерв при PM-фичах — разумный compromise |
| dhtmlx / bryntum / syncfusion | ✅ Стоп по лицензии для проприетарной CRM |

Отдельный путь «custom SVG / visx» из roadmap **не вынесен** в таблицу — но он по сути схлопнут с «Кастом (CSS-grid)»; для ADR это ок, v0 уже выбрал grid.

### 3. Зависимости сейчас не драйвер — факт из схемы

В `tasks` нет `predecessor_id` / `task_dependencies` (roadmap §9.3 — v2). Главное преимущество тяжёлых либ (стрелки, critical path) **не реализуемо** без миграции — аргумент спайка силён.

### 4. @dnd-kit — реально в репо, но с оговоркой

Подтверждено: `package.json` (`@dnd-kit/core` 6.3, sortable 10), используется в `ProjectBoard.tsx` и `KanbanBoard.tsx` (`DndContext`, `useSortable` на `TaskCard`).

Спайк честно не обещает «drag из коробки на Gantt»: риск «grid-бары vs sortable-колонки» вынесен в §Риски. Текущий dnd-kit-паттерн — **перестановка в списке**, не drag по оси дат. Для resize/move по timeline почти наверняка понадобятся **custom sensors / pointer handlers**, а dnd-kit — только как часть инфраструктуры. Формулировка «единственная интерактивная фича v1 — инкремент на существующем стеке» — **чуть оптимистична**, но направление верное (не тащить вторую drag-либу).

### 5. v0 как proof-of-concept

- `GanttTimeline.tsx` — **153 строки** (спайк: «~150» — точно).
- Коммит в спайке `3f78623` существует; в текущей истории tip — `685864d` (тот же message, вероятно amend/rebase). Для ссылки лучше **последний** hash или «main, S-GANTT-V0-1».
- Работает: `mskDateKey`, `buildDays` UTC-полдень, fallback deadline, таб на `ProjectDetail` — спайк корректно ссылается на доказанный подход.

### 6. Sequencing «zoom → drag»

Рекомендация в конце (S-GANTT-VIEW-1 начать с zoom, drag отдельным спринтом) — **правильная декомпозиция**: zoom не блокируется hit-test resize, даёт ценность на планёрках раньше.

### 7. Триггер пересмотра

«Зависимости + critical path → переоценить SVAR» — зрелый exit criteria. Совпадает с `CRM-ROADMAP` §9.3 и `PM-GROWTH-ZONES.md`.

## Рекомендации по доработке документа (опционально)

### 1. Явная ссылка на roadmap §9.2

Одна строка: «Продуктовый scope v1 — §9.2 CRM-ROADMAP; drag — v2 по roadmap, отдельный спринт по технике спайка».

### 2. Уточнить оценку v1

«1–2 спринта» для zoom+drag+polish — реалистично для **техники**. С учётом swimlane + milestones + filters из §9.2 — скорее **2–3 спринта** на read-only VIEW-1, если не резать фильтры.

### 3. Виртуализация — добавить числовой триггер

Сейчас «при очень широких диапазонах». Из v0: `buildDays` без капа → год = 365 колонок. Имеет смысл зафиксировать: **>180 дней на day-zoom → предупреждение в UI; виртуализация при >365 или при >N задач**.

### 4. Milestone на Gantt — не ждать либу

`is_milestone` уже в схеме; на кастоме ромб = смена формы бара (`rotate-45` / SVG) + токен `bg-accent` — проще, чем интеграция frappe milestones.

### 5. Источники

Ссылки на npm/github — достаточно для ADR. Для гейта можно добавить дату проверки downloads SVAR/@wamra (цифра «576 dl/нед» не верифицировалась в этом ревью — не оспариваю, но при пересмотре через год обновить).

## Риски — дополнение к спайку

| Риск | Комментарий ревью |
|------|-------------------|
| dnd-kit ≠ date-axis DnD | Для drag-спринта заложить **полдня spike** внутри задачи: pointer-based resize vs DndContext. Не копировать паттерн `useSortable` с доски. |
| Zoom month + MSK | Бакеты неделя/месяц должны использовать ту же дисциплину, что `mskDateKey` / UTC-полдень — иначе бары «выпадут» из бакета на границе месяца. |
| Группировка по фазе | На delivery `column_id` → фаза СДР; на client/internal — плоский список или группа «Без колонки». Roadmap подразумевает swimlane — это **отдельная модель строк**, не только zoom. |
| Performance без виртуализации | Week/month zoom снимает проблему для планёрок; day-zoom оставить как в v0 с горизонтальным скроллом. |
