## Что сделано особенно хорошо

### 1. Правильный уровень абстракции

Разделение на два ортогональных уровня (из `delivery-process-DO.md`) перенесено точно:

| Уровень в 1С:ДО | Аналог в CRM | Зачем |
|---|---|---|
| 4 состояния запуска | `stage_id` на project-пайплайне | «Где проект в целом» |
| Дерево задач СДР | PCT-1 доска | «Что конкретно делаем» |

Это снимает типичную ошибку — пытаться впихнуть нелинейный план в линейную воронку.

### 2. Расширение `projects`, а не новая таблица

Решение обосновано. Уже есть:

- дискриминатор `type` с CHECK-инвариантами (миграция 032)
- PCT-1 (`project_columns` + `tasks.column_id`)
- org-scoped RLS через `user_role()`
- паттерн spawn из `convert_lead()`

Третий тип `delivery` — естественное продолжение, а не костыль.

### 3. Индустриальная валидация

Паттерн Deal → Delivery Project (1:N) совпадает с Salesforce PSA, HubSpot Projects и фактическим процессом в 1С:ДО. Модель `parent_deal_id` + `delivery_kind` (launch/experiment) покрывает реальный сценарий «сначала пилот, потом полный запуск».

### 4. Фазирование по ROI

Roadmap P1→P4 реалистичен:

- **P1** даёт видимость сразу (сейчас после won — «чёрная дыра»)
- **P2** — операционная ценность (шаблоны)
- **P4** (1С-синк) правильно отложен с высоким риском

### 5. Шаблоны «супермножество + выключить»

Паттерн `default_enabled=false` для «НЕ ТРЕБУЕТСЯ!» — прагматичный и совпадает с реальной практикой копирования шаблонов в ДО.

## Зоны риска и пробелы

### 1. Конфликт с `seed_project_columns` (критично для P1/P2)

Сейчас триггер `trg_zz_seed_columns` при **любом** INSERT в `projects` создаёт 4 generic-колонки: Бэклог / В работе / Ожидание / Готово.

Для delivery нужны 4 **фазы СДР** из шаблона. Без доработки spawn создаст **8 колонок** (4 дефолтных + 4 из шаблона) или semantic mismatch.

**Рекомендация:** сделать `seed_project_columns` type-aware:

```sql
IF NEW.type = 'delivery' THEN RETURN NEW; END IF;  -- колонки создаёт spawn
```

Или отключить триггер для delivery и копировать колонки только из шаблона в `spawn_delivery_project`.

### 2. Семантика колонок PCT-1 меняется для delivery

Сейчас в PCT-1: **колонка = swimlane задачи** (backlog/started/paused/done), `lane` — производное.

В концепции для delivery: **колонка = фаза СДР**, а состояние задачи (Не начата/В работе/Просрочена/Готово) — через `lane`.

Это работает, но требует:

- отдельного UI для delivery-борда (фазы как колонки, статус задачи — badge/подколонка, не swimlane)
- явного контракта: для `type='delivery'` колонки ≠ категории lane в том же смысле, что у internal

Документ описывает это концептуально, но не фиксирует **UI-контракт** — стоит добавить в P2.

### 3. CHECK-инвариант `projects_type_pipeline_chk` нужно расширить

Сейчас (032):

```sql
(type = 'client'   AND pipeline_id/stage_id/direction NOT NULL)
OR
(type = 'internal' AND pipeline_id/stage_id/direction IS NULL)
```

Для `delivery` нужна третья ветка — по сути как `client`, но на **project-пайплайне** (4 стадии), не deal-воронке. Без этого spawn технически невозможен или нарушит инвариант.

Предлагаемая ветка:

```sql
OR (type = 'delivery' AND pipeline_id IS NOT NULL AND stage_id IS NOT NULL AND direction IS NOT NULL)
```

Плюс отдельный CHECK: `delivery` не может иметь `status IN ('won','lost')` — это статусы сделки, не delivery.

### 4. Денормализация `progress_done` / `progress_total`

Кэш прогресса — нормально для карточки и списка, но нужен механизм обновления:

- триггер на `tasks` (INSERT/UPDATE/DELETE) для delivery-проектов
- или вычисление в view/RPC при чтении (проще, без drift)

Без этого поля быстро устареют — особенно если план редактируют в P2.

### 5. Двойная ось `stage_id` + `status`

Аналогия со сделками (`stage_id` + `status=won/lost`) здравая. Но нужны явные правила:

- «Завершён» на пайплайне ↔ `status='completed'` — синхронизировать вручную, триггером или одним полем?
- «Приостановлен» — это `status='on_hold'` при любом `stage_id`?

Открытый вопрос №1 в документе — не косметический; от него зависит UX канбана из 4 колонок в P1.

### 6. `parent_deal_id ON DELETE SET NULL`

При удалении сделки delivery-проект станет сиротой с нарушением `projects_delivery_chk` (delivery без parent).

Лучше `ON DELETE RESTRICT` или soft-delete сделок. SET NULL опасен.

### 7. Команда проекта (роли из 1С:ДО)

В ДО обязательны 3 роли: Менеджер, Внедренец, Монтажник. В модели этого нет — ни в P1, ни в roadmap.

Для P1 можно жить с `owner_id`, но для операционной ценности P2 стоит заложить `project_members` или переиспользовать существующий механизм назначения.

### 8. Оценка «~1 спринт» на P1 — оптимистична

P1 включает:

- миграцию + новый CHECK + RPC spawn
- пересоздание project-пайплайнов
- обновление validators (`project.ts`), hooks (`use-projects.ts`), UI-гейтинг по `type`
- новый раздел навигации + канбан 4 состояний
- кнопку spawn на won-сделке

Реалистичнее **1–1.5 спринта**, если делать качественно с тестами и RLS-smoke.

## Рекомендации по открытым вопросам

| # | Вопрос | Рекомендация |
|---|---|---|
| 1 | Приостановлен/Отменён | `on_hold` / `cancelled` в `status`, не в пайплайне. Пайплайн = 4 состояния из 1С, health — отдельно |
| 2 | ERP vs IIoT | Отдельные шаблоны + `direction`-фильтр на project-пайплайне. Одна модель, разный контент |
| 3 | Навигация | «Проекты» = delivery + internal, фильтр по типу — согласен с документом. Сделки остаются в «Сделках» |
| 4 | 1:N сделка→проекты | Модель верна. В UI на карточке сделки — список delivery-проектов |
