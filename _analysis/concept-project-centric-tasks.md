# Концепт: Project-centric Tasks (эволюция модуля задач)

Дата: 2026-07-08. Направление: сместить ось задач с «клиент/звонок/событие» на ПРОЕКТ.
Требования Олега: (1) внутренние проекты без клиента; (2) кастомные этапы канбана
(имя + количество колонок) под каждый проект.
Статус: концепт для обсуждения, до sprint prompt.

## Текущее состояние (факты из кода)

- `/tasks` = один **глобальный** Kanban по `lane` (`useTasksByLane`). Задачи всех проектов
  и клиентов свалены в общий борд с фиксированными колонками (enum `TaskLane`).
- `tasks`: `lane`, `priority`, `assignee`, `project_id` (nullable), `company_id`, `contact_id`,
  `deadline`, `sort_order`. Проект — просто поле связи, НЕ ось организации.
- `projects` (=сделки): `company_id`/`contact_id` — **nullable** (клиент уже не обязателен
  на уровне колонок), `stage_id` — **NOT NULL** (каждый проект обязан быть в sales-воронке),
  `status`, `budget`, `deadline`, `phase_group` на стадиях (треки подготовка/эксперимент/проект).
- Sales-воронка обвешана логикой: гейты S27, автоматизация переходов S29, probability,
  lost-handling — всё завязано на `stage_id → pipeline_stages`.

## КРИТИЧНО: два разных «этапа», не путать

Есть ДВА ортогональных понятия «стадия», и их нельзя смешивать в одной колонке:

1. **Стадия сделки** (`stage_id → pipeline_stages`) — где сделка в ВОРОНКЕ ПРОДАЖ:
   Лид → Квалификация → Подготовка КП → Won/Lost. Гейты, автоматизация, probability.
   Отвечает на вопрос «где мы в продаже».
2. **Колонка канбана задач** (НОВОЕ) — где конкретная ЗАДАЧА в ИСПОЛНЕНИИ работы:
   Бэклог → В работе → На проверке → Готово (или кастом под проект).
   Отвечает на вопрос «где мы в работе».

«Кастомные этапы канбана под проект» = **пункт 2** (доска исполнения), НЕ sales-воронка.
Их надо держать в РАЗНЫХ таблицах. Перегрузить `stage_id`/`pipeline_stages` под доску задач —
значит столкнуться с гейтами и автоматизацией S29. Не делаем.

Следствие:
- Клиентский проект имеет ОБА: стадию сделки (воронка) + колонки доски задач (исполнение).
- Внутренний проект имеет ТОЛЬКО колонки доски задач (в воронке продаж его нет).

## Модель данных (delta, аддитивно)

### Внутренние проекты
- `projects.type` text ('client' | 'internal'), default 'client'. Явный тип, не «угадывать по
  пустому company_id».
- `projects.stage_id` → сделать **nullable**. Внутренний проект (type='internal') имеет
  `stage_id = null` — он не в воронке продаж. Миграция NOT NULL→nullable безопасна (аддитивна),
  но код в куче мест ждёт stage_id → внутренние проекты должны ПРОПУСКАТЬ всю pipeline-логику.
- **Гейты S27, автоматизация S29 (stage-change), probability, lost-handling — только для
  type='client'.** Для internal — пропускать (иначе движок продаж выстрелит на «допилить CRM»).

### Кастомные колонки канбана (per-project)
- `project_columns` { id, org_id, project_id, name, position, category, wip_limit? }.
  Колонки — ДАННЫЕ, не enum. Имя и количество — свои у каждого проекта.
- `category` text ('backlog' | 'started' | 'done' | 'cancelled') — нормализующий класс.
  Зачем: у проектов разные ИМЕНА колонок, но глобальный личный борд и метрики должны
  сводить их к единому Todo/Doing/Done. Это паттерн Linear (custom statuses + category).
  Чат про Work OS это и советовал («issue_status как таблица с type, не enum») — здесь
  конкретное обоснование: без category кастомные колонки ломают кросс-проектные виды.
- `tasks.column_id` uuid nullable FK → project_columns. Задача живёт в колонке своего проекта.
- При создании проекта — **сидировать дефолтные колонки** (Бэклог/В работе/Готово), чтобы
  борд не был пустым. Для internal — тоже.

### Секции/фазы внутри проекта (Фаза 2, опц.)
- Для длинных внедрений: `project_sections` (или переиспользовать phase_group-идею) —
  группировка задач по этапу проекта (Обследование/Настройка/Тестирование/ОЭ). Ортогонально
  колонкам: секция = «какой этап проекта», колонка = «статус задачи». Решим на Фазе 2.

### Что с `lane` (backward-compat — важно)
- `lane` НЕ выпиливать сразу. На него завязаны: глобальный личный борд, автоматизация S29
  (`create_task` пишет `lane` по whitelist), тип `TaskLane` по всему коду.
- Migration-safety: аддитивно добавить `project_columns` + `column_id`, бэкфилл (существующие
  задачи → дефолтные колонки их проекта по маппингу lane→category), dual-read, и только
  после проверки — решать судьбу `lane`. Скорее всего `lane` останется как быстрый статус
  глобального личного борда, а `column_id` — для проектных досок (два разреза, см. ниже).

## Два разреза, оба остаются

- **Проектный борд** (новое, ядро): открыл проект → доска задач с кастомными колонками,
  создаёшь/двигаешь/группируешь. Управление работой по проекту.
- **Личный борд** (существующий `/tasks`): «мои задачи» поверх всех проектов, группировка
  по `category` (нормализованный статус) или по проекту. Триаж, не исполнение.
- Не конкуренты — паттерн Linear «My Issues vs Project view». Существующий борд не выкидываем.

## Фазирование (в твоём стиле — каждая фаза ценна сама)

| Фаза | Скоуп | Оценка | Риск |
|---|---|---|---|
| **P1 — Ось + кастомные колонки + internal-проекты** | project.type, nullable stage_id (+ exclude pipeline для internal), project_columns + task.column_id + category, проектный борд, сид дефолтных колонок, личный борд на category | ~1 спринт (тяжелее полишей — миграция + KanbanBoard рефактор + S29 совместимость) | Средний-высокий |
| **P2 — Глубина исполнения** | секции/фазы внутри проекта, подзадачи (parent_task_id), зависимости (blocks/blocked-by) | ~1 спринт | Средний |
| **P3 — Повторяемость (домен-киллер)** | шаблоны проектов (playbook внедрения → авто-генерация колонок+секций+задач), вехи (milestones) | ~0.7 спринта | Низкий (аддитивно поверх P1/P2) |

P1 — фундамент, делается первым. P3 (шаблоны маркировочного внедрения) — то, что реально
экономит твои часы на каждом клиенте, но требует P1/P2 под собой.

## Главные риски / на что смотреть

1. **Не перегрузить `stage_id` под доску задач** — две таблицы, два понятия (см. выше).
   Это риск №1: если смешать, столкнёшься с гейтами/автоматизацией.
2. **Internal-проекты обязаны выпадать из sales-машинерии** — гейты, S29, probability, воронка.
   Проверить каждую точку, где код читает stage_id, на предмет null (internal).
3. **`lane`-совместимость** — S29 `create_task` пишет lane; не сломать автоматизацию.
   Аддитивная миграция + dual-read, не rip-and-replace.
4. **KanbanBoard рефактор** — сейчас drag-drop по фиксированным lane; переезд на динамические
   колонки — самый большой кусок фронта P1. `dnd` логика (`useTasksByLane`) переписывается на
   `column_id`.
5. Это заметно тяжелее последних полишей — полноценный спринт с миграцией, не правка модалки.

## Открытые вопросы (для P1 sprint prompt)

1. Внутренние и клиентские проекты — в одном списке `/projects` с фильтром по типу, или
   разные разделы навигации? (Влияет на UX, не на модель.)
2. Дефолтный набор колонок нового проекта — фикс (Бэклог/В работе/Готово) или свой для
   client vs internal? (P3-шаблоны это обобщат, но дефолт нужен уже в P1.)
3. Задача без проекта (`project_id = null`) — куда её колонка? Личный борд остаётся на `lane`
   для беспроектных задач — это ещё один аргумент не выпиливать `lane`.
