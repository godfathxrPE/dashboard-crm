# AI-блок (AI Hub) — концепт

Дата: 2026-07-07. Статус: черновик для обсуждения, до sprint prompt.
База: S28 (`ai-summarize`, 028_ai_summary.sql) + паттерны S29 (runs, идемпотентность).

## 1. Главное решение: не «чат», а библиотека воркфлоу

Самая частая ошибка AI-блоков в CRM — сделать «окно с чатом». Оно демо-эффектно и бесполезно в работе: результат не структурирован, никуда не привязан, не воспроизводим.

Правильная рамка: **AI Hub = набор пресетов (воркфлоу) с типизированным входом и типизированным выходом, привязанным к сущности CRM.**

```
Пресет = { вход (транскрипт/сущность) → фикс. промпт → structured output → артефакт в CRM }
```

Каждый прогон: воспроизводим, логируем, привязан к сделке/звонку, имеет стоимость и автора.

## 2. Где живут промпты: код, не БД

Соблазн — таблица `ai_presets` с редактируемым `system_prompt`. **Не делать.** Причины:

1. S28 уже выбрал верную модель безопасности: системный промпт фиксирован в коде, пользовательские данные — только в `<data>`. Редактируемый промпт в БД ломает весь injection-контур.
2. Промпт без своей tool-схемы (structured output) бесполезен — а схему в БД не положишь.
3. Качество промпта — это QA-артефакт: его надо версионировать в git, а не править в проде.

Итог: **реестр пресетов в коде** edge function (`PRESETS: Record<string, Preset>`), в БД — только включённость и настройки.

```ts
type Preset = {
  key: string;                // 'meeting_protocol' | 'analytic_note' | 'spin_review' | ...
  title: string;
  description: string;
  input: 'transcript' | 'entity' | 'transcript+entity';
  entityTypes: ('call' | 'meeting' | 'project')[];
  model: string;              // haiku для дешёвых, sonnet для сложных
  maxInputChars: number;      // у транскриптов свой лимит, не 8000 из S28
  system: string;             // фиксированный, с анти-injection преамбулой
  tool: AnthropicTool;        // structured output схема
  promptVersion: number;      // пишем в meta каждого прогона
};
```

Кастомизация без риска: опциональное поле `org_ai_settings.extra_context` (описание компании, специфика — «мы продаём внедрение маркировки, типовой чек 1–5 млн»), которое подставляется **как данные** в `<data kind="org_context">`, не в system.

## 3. Схема данных

### 3.1 `transcripts` — транскрипт как самостоятельная сущность

Не поле на `calls` — отдельная таблица. Причины: один звонок может иметь несколько версий транскрипта; транскрипт нужен и встречам; на него ссылаются несколько прогонов (протокол + SPIN по одному транскрипту — 2 прогона, 1 транскрипт).

```sql
CREATE TABLE transcripts (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id       uuid NOT NULL,             -- trg_set_org_id, RLS как у остальных
  entity_type  text NOT NULL CHECK (entity_type IN ('call','meeting')),
  entity_id    uuid NOT NULL,
  source       text NOT NULL CHECK (source IN ('paste','file')),  -- 'stt' в будущем
  content      text,                      -- если влезает (< ~200К символов — всегда)
  storage_path text,                      -- оригинал файла в Storage (private bucket)
  char_count   int NOT NULL,
  created_by   uuid NOT NULL,
  created_at   timestamptz NOT NULL DEFAULT now()
);
```

v1: paste в textarea. v1.5: upload .txt/.docx/.vtt (парсинг на клиенте или в function). STT (аудио → текст) — сознательно out of scope, но `source` уже готов.

⚠️ PII: транскрипты — это персональные данные клиентов (имена, телефоны, коммерческие условия). Bucket строго private, доступ через RLS/signed URL, в activity_log — только факт, не содержимое. Это же аргумент против отправки транскриптов куда-либо кроме Anthropic API.

**Retention:** хранить транскрипты вечно — риск (152-ФЗ) и мусор. Политика: исходники в Storage чистятся через N дней (default 90, настройка org), `content` живёт дольше — он нужен как вход QA-датасета (см. rating в 3.2); чистить его вместе с прогонами или по явному запросу юзера («удалить транскрипт» → каскадом пометить runs как input-deleted). Нативного TTL у Supabase Storage нет — это pg_cron / scheduled function. В MVP достаточно задокументировать политику и колонки, сам cron — S-AI-2.

### 3.2 `ai_runs` — журнал прогонов (клон паттерна automation_runs)

```sql
CREATE TABLE ai_runs (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id        uuid NOT NULL,
  preset_key    text NOT NULL,
  entity_type   text,
  entity_id     uuid,
  transcript_id uuid REFERENCES transcripts(id),  -- NOT NULL в MVP (все пресеты v1 идут от транскрипта);
                                                  -- nullable станет в S-AI-3 вместе с entity-пресетами
  status        text NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending','running','done','error')),
  result        jsonb,                    -- structured output пресета
  error         text,
  model         text,
  prompt_version int,
  input_tokens  int,                      -- из usage ответа API, не оценка
  output_tokens int,
  duration_ms   int,
  rating        smallint,                 -- 1 / -1, фидбек юзера
  feedback_note text,                     -- опционально при 👎: «что не так»
  created_by    uuid NOT NULL,
  created_at    timestamptz NOT NULL DEFAULT now(),
  finished_at   timestamptz
);
```

Зачем `input_tokens`/`output_tokens` с первого дня: это учёт денег. Страница расхода по org = `SUM(tokens) * price(model)`. Без этого через месяц будет «куда делись кредиты» без ответа. Сюда же позже прикручивается квота: `N прогонов/день на юзера` — тривиальный check перед стартом.

`rating` — 👍/👎 на результате, при 👎 — необязательное поле «что не так» (`feedback_note`). Отдельная таблица «плохих прогонов» не нужна: `ai_runs` уже хранит и вход (`transcript_id` → content), и выход (`result`), и версию промпта. QA-датасет для регрессии новых версий промптов — это просто `SELECT ... WHERE rating = -1` с джойном на транскрипт. Ничего копировать не надо, надо только не удалять транскрипты раньше их прогонов (см. retention в 3.1).

### 3.3 Что НЕ делаем

- `ai_summary` на calls/meetings из S28 не трогаем — живёт параллельно как «быстрое саммари по заметкам». AI Hub — про тяжёлые прогоны по транскриптам.
- Никакого векторного поиска/RAG в v1. История прогонов + привязка к сущности покрывает 90% ценности.

## 4. Пресеты: состав

### v1 (MVP, один спринт)

| Пресет | Вход | Модель | Выход (structured) |
|---|---|---|---|
| **Протокол встречи** | transcript | sonnet | участники, повестка, обсуждено, решения, **action_items[{что, кто, срок}]**, открытые вопросы |
| **Аналитическая записка** | transcript + entity | sonnet | ситуация клиента, потребности/боли, стейкхолдеры и роли, риски сделки, рекомендации, аргументы для КП. Анти-галлюцинация: каждое утверждение о боли/риске — с цитатой-основанием из транскрипта (`{claim, quote}`) |
| **SPIN-разбор звонка** | transcript | sonnet | счёт по S/P/I/N (сколько вопросов какого типа, с цитатами), что упущено (какие implication/need-payoff не заданы), 3 конкретных вопроса для следующего звонка, общая оценка 1–10 с обоснованием |

Контент промптов не с нуля: протокол — из skill `protocol-design-is` (структура секций), SPIN — из `spin-sales-analyst` (методология Рекхема, критерии оценки). Это готовые, обкатанные тексты — задача только адаптировать под structured output.

### v2 (после обкатки — каждый пресет дёшев, инфраструктура общая)

- **Подготовка к звонку** — вход не транскрипт, а сделка: история активностей, прошлые протоколы, стадия → бриф «что мы знаем, что выяснить, какие вопросы задать». Замыкает цикл: разбор прошлого звонка → подготовка следующего.
- **Черновик follow-up письма** по итогам встречи (протокол уже есть → письмо из него почти бесплатно, haiku).
- **Извлечение полей** — из транскрипта вытащить бюджет, сроки, ЛПР, конкурентов → **предложить** обновить сделку (см. 4.1).
- **Risk-скан сделки** — по всей истории сделки: сигналы затухания, несоответствие стадии и активности.

### 4.1 Killer-фича MVP: action items → задачи

Протокол возвращает `action_items[]`. Рядом с каждым — кнопка «Создать задачу»: prefill формы задачи (title, due_date, company/project из контекста), юзер подтверждает.

Принцип жёсткий: **AI никогда не пишет в CRM сам — только предлагает, юзер подтверждает.** (Единственное исключение — собственные таблицы `ai_runs`/`transcripts`.) Это одновременно:
- защита от injection в транскрипте («создай задачу "перевести деньги"...» — юзер увидит и не подтвердит);
- защита от галлюцинаций в данных сделки;
- доверие пользователей — AI-фичи умирают, когда начинают молча портить данные.

То же для «извлечения полей» в v2: diff-превью «бюджет: пусто → 2,5 млн» с кнопками принять/отклонить, по образцу convert_lead.

## 5. Исполнение: асинхронность

Транскрипт 40–60 мин ≈ 50–100К символов ≈ 15–30К токенов входа, генерация 30–90 сек. Синхронный fetch из UI не годится (таймауты, закрытая вкладка = потерянный результат).

Схема — одна generic edge function `ai-run`:

```
POST /ai-run { preset_key, entity_type, entity_id, transcript_id }
  1. Валидация + RLS-загрузка (паттерн S28: JWT юзера, не нашлось → 404)
  2. INSERT ai_runs (status='pending') → сразу вернуть { run_id }   ← ответ юзеру < 1 сек
  3. EdgeRuntime.waitUntil(process(run_id)):
       status='running' → Claude API → status='done', result, usage
       любая ошибка → status='error', error   (прогон никогда не виснет в running)
```

Клиент: `useAiRun(runId)` — Realtime-подписка на строку `ai_runs` (fallback: поллинг 3 сек). UI показывает pending → running → результат, можно уйти со страницы и вернуться.

Нюансы:
- **Wall-clock лимит** edge functions (~150–400 сек в зависимости от плана) — одного прогона хватает с запасом, но retry внутри function не делаем; ошибка → status='error', кнопка «Повторить» создаёт новый run.
- **Конкурентность:** partial unique index на `(transcript_id, preset_key) WHERE status IN ('pending','running')` — двойной клик не создаст второй прогон, но два разных транскрипта одной сделки (звонок с ЛПР + звонок с техдиром) друг друга не блокируют. Ключ прогона — транскрипт, не сущность. Когда в S-AI-3 появятся entity-пресеты без транскрипта — добавится второй partial index `(entity_type, entity_id, preset_key) WHERE transcript_id IS NULL AND status IN ('pending','running')`. Идемпотентность по паттерну S29.1.
- **Таймаут-страховка:** `pending`/`running` старше 10 мин считаем протухшим (UI показывает «зависший, повторить»); можно добить cron-cleanup позже.

Security-контур — целиком наследуется от S28 (промпт в коде, `<data>`, forced tool_use, ключ в secrets, JWT+RLS). Единственное отличие: `MAX_INPUT_CHARS` per preset (для транскриптов ~120К символов вместо 8К), обрезка с честной пометкой в result.meta.

## 6. UI: две точки входа

### 6.1 Основная — вкладка/секция «AI» на карточке звонка и встречи

Рабочий сценарий менеджера: закончил звонок → открыл его карточку → вставил транскрипт → нажал «Протокол» и «SPIN-разбор». Всё в контексте сущности, ничего искать не надо.

Состав: зона транскрипта (paste/файл, счётчик символов, «≈ N ₽ за прогон»), кнопки пресетов, лента прогонов этой сущности (свежий сверху, статус-чип, развёрнутый результат, 👍/👎, экспорт).

### 6.2 Вторичная — страница `/ai` (хаб)

Каталог пресетов с описаниями + история всех прогонов org (фильтр по пресету/юзеру/сущности) + расход токенов за месяц. Для руководителя и для «найти вчерашний протокол». Не строить в MVP сложнее, чем список.

Позже: пункт в Cmd+K (S31) — «AI: протокол по последнему звонку с …».

### 6.3 Рендер результата

Каждый пресет — свой renderer по своей схеме (протокол ≠ SPIN-разбор). Общие правила: только текст (никакого markdown→HTML из модели — XSS/injection-гигиена, как в S28), кнопка «Копировать», для протокола — **экспорт в .docx по протокол-дизайн-системе** (у нас готовый skill `protocol-design-is`: фирменный протокол из CRM в один клик — сильный wow для демо).

## 7. Экономика (ESTIMATED)

Прогон протокола: ~25К токенов in + ~2К out.

| Модель | Цена прогона | Комментарий |
|---|---|---|
| Sonnet | ~$0.10–0.12 (≈10 ₽) | протокол, записка, SPIN |
| Haiku | ~$0.01 (≈1 ₽) | follow-up, извлечение полей |

При 10 звонках/день × 2 прогона ≈ $2–2.5/день на активного менеджера. Не страшно, но не бесплатно → учёт токенов и мягкая квота с первого дня (см. 3.2).

**Prompt caching — почему «почти не помогает» (разобрано, решение осознанное).** Сценарий «протокол + SPIN по одному транскрипту подряд» выглядит идеальным для кэша, но кэш Anthropic — это префикс-кэш со строгим порядком: `tools → system → messages`. У каждого пресета свои `tools` (своя output-схема) и свой `system` → префикс расходится в первом же блоке, транскрипт в messages не переиспользуется. Плюс разный форсированный `tool_choice` сам по себе инвалидирует кэш messages. Заставить кэш работать можно, только унифицировав промпт-архитектуру: общий system, общий пул tools всех пресетов, преset-инструкция после транскрипта — это размывает качество и security-контур ради ~3–5 ₽ экономии на паре прогонов при TTL кэша 5 минут. Не делаем. Кэш станет актуален, если появится пресет с большим статичным контекстом (например, база знаний org в system) — тогда вернуться.

## 8. Чего не хватало в исходной идее — сводка

1. **Транскрипт как сущность**, а не одноразовый ввод: один транскрипт → несколько прогонов, история, PII-контур.
2. **AI предлагает — юзер подтверждает**: action items → задачи, поля → diff-превью. Это и есть превращение «генератора текстов» в CRM-фичу.
3. **Учёт токенов и фидбек (rating)** с первого дня — иначе слепая зона по деньгам и качеству.
4. **Промпты в коде с версией**, не в БД — безопасность + QA. Кастомизация только через org_context-данные.
5. **Замыкание цикла** (v2): разбор звонка → подготовка к следующему. Ретроспектива без проспективы — половина ценности.
6. **Экспорт протокола в .docx** по готовой дизайн-системе — дешёвый и очень заметный полиш.
7. **Идемпотентность и зависшие прогоны** — не забыть, вы это уже проходили в S29.1.

## 9. Фазирование

| Фаза | Скоуп | Оценка |
|---|---|---|
| **S-AI-1 (MVP)** | transcripts (paste) + ai_runs + edge `ai-run` + 3 пресета + вкладка AI на call/meeting + action items→задачи + учёт токенов | 1 спринт, масштаб S29 |
| **S-AI-2** | файл-upload, страница /ai, docx-экспорт протокола, 👍/👎, follow-up-пресет | ~0.5 спринта |
| **S-AI-3** | подготовка к звонку, извлечение полей с diff-превью, квоты | ~0.5–1 спринт |

Блокер по-прежнему один: кредиты Anthropic. Injection-тест из бэклога S28 — обязательный гейт S-AI-1 (транскрипт — идеальный вектор атаки: длинный, недоверенный, целиком в промпте).

## 10. Открытые вопросы → решения (после ревью 2026-07-07)

1. **Источник транскриптов: paste сейчас, структура заложена.** v1 принимает плоский текст «отовсюду» (`source='paste'`), роли спикеров модель вычитывает из контекста сама (обычно транскрипты и так содержат «Имя:» в начале реплик). Поле `source` и схема готовы к структурированным форматам (VTT/диаризация с таймкодами) — парсер спикеров добавляется в S-AI-2 без миграций, когда определится реальный сервис записи.
2. **Видимость прогонов — по RLS сущности.** Протокол звонка — артефакт компании, не личная заметка менеджера: кто видит звонок, тот видит его транскрипты и прогоны. Консистентно с остальной CRM, отдельных политик не плодим.
3. **Протокол без транскрипта — НЕ в AI Hub.** Резюме по коротким заметкам — епархия `ai_summary` из S28 (при желании расширить его схему). AI Hub — про перемалывание больших сырых текстов в структуру. Следствие для схемы: в MVP `transcript_id` в `ai_runs` — NOT NULL, уникальный ключ `(transcript_id, preset_key)` без дырок; nullable появится только в S-AI-3 с entity-пресетами.

## 11. Отклонённые предложения (чтобы не возвращаться)

- **Prompt caching между пресетами** — см. §7: ломается о разные tools/system/tool_choice, унификация промптов ради копеек не стоит размытия security-контура.
- **Отдельная таблица «плохих прогонов»** — избыточна, `ai_runs + rating + feedback_note` уже полный QA-датасет (§3.2).
- **Мульти-агентная валидация** (второй прогон-верификатор для аналитической записки) — идея здравая, но х2 к стоимости и сложности. Дешёвая альтернатива уже в промпте: требовать цитату-основание для каждого утверждения о «болях»/рисках (schema: `{claim, quote}`) — галлюцинации видны глазами без второго прогона. Вернуться, только если фидбек покажет систематические выдумки.
- **Коллекция методологий (Чалдини, 5P и т.п.)** — архитектура и так позволяет добавить пресет за час; наполнять каталог надо от спроса пользователей, а не от списка фреймворков.
