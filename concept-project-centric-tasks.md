Требования Олега: (1) внутренние проекты без клиента; (2) кастомные этапы канбана

## Текущее состояние (факты, ПРОВЕРЕНО)

- `/tasks` = один **глобальный** Kanban по `lane` (`useTasksByLane` в `use-tasks.ts`). Задачи всех
  проектов и клиентов свалены в общий борд с фиксированными колонками (enum `TaskLane`,
  используется в 8 файлах: KanbanBoard, LaneColumn, AccordionLane, TaskQuickAdd,
  constants/automation, timeline/adapters, types, hooks).
- `tasks`: `lane`, `priority`, `assigned_to`, `project_id` (nullable), `company_id`, `contact_id`,
  `deadline`, `sort_order`. Проект — просто поле связи, НЕ ось организации.
- `projects` (=сделки), **нуллабельность из живой БД**:
  - `company_id` / `contact_id` — nullable (клиент не обязателен на уровне колонок);
  - `stage_id` — **NOT NULL** → pipeline_stages;
  - `pipeline_id` — **NOT NULL** (v1 это поле пропустил — internal-проектам придётся
    занулять ОБА);
  - `direction` — **NOT NULL** enum `direction_t` (`erp`/`iiot`) — internal туда не влезает;
  - `stage` — legacy, nullable, но с **DEFAULT `'new_lead'`** — производное зеркало
    `stage_id` через `mapToLegacyStage()` (см. SPRINT-stage-deprecation.md); дефолт при
    insert internal-проекта молча запишет 'new_lead', если не убрать;
  - `status` — NOT NULL DEFAULT `'open'` (`open`/`won`/`lost`/`on_hold`) — won/lost
    это терминология сделки.
- Sales-воронка обвешана логикой: гейты S27, автоматизация S29, probability, lost-handling,
  sync-триггер `sync_project_stage()` (019: stage_id → status/actual_close_date/stage_entered_at).
- **Хорошая новость №1**: `sync_project_stage()` уже NULL-safe — при `stage_id IS NULL`
  сразу `RETURN NEW`. Прецедент проектов вне воронки в триггерах частично заложен.
- **Хорошая новость №2**: S29 `run_stage_automations()` (AFTER UPDATE ON projects) матчит
  правила по `(trigger_config->>'stage_id')::uuid = NEW.stage_id`. Для internal
  (`stage_id = NULL`) сравнение с NULL не матчится никогда → автоматизация выключается
  на уровне БД сама. Отключать её надо только во фронте (UI стадий/гейтов).

## КРИТИЧНО: два разных «этапа», не путать

Есть ДВА ортогональных понятия «стадия», и их нельзя смешивать в одной колонке:

1. **Стадия сделки** (`stage_id → pipeline_stages`) — где сделка в ВОРОНКЕ ПРОДАЖ:
   Лид → Квалификация → Подготовка КП → Won/Lost. Гейты, автоматизация, probability.
   Отвечает на вопрос «где мы в продаже».
2. **Колонка канбана задач** (НОВОЕ) — где конкретная ЗАДАЧА в ИСПОЛНЕНИИ работы:
   Бэклог → В работе → На проверке → Готово (или кастом под проект).
   Отвечает на вопрос «где мы в работе».

«Кастомные этапы канбана под проект» = **пункт 2** (доска исполнения), НЕ sales-воронка.
Разные таблицы. Перегрузить `stage_id`/`pipeline_stages` под доску задач — столкнуться
с гейтами S27 и автоматизацией S29. Не делаем.

Следствие:
- Клиентский проект имеет ОБА: стадию сделки (воронка) + колонки доски задач (исполнение).
- Внутренний проект имеет ТОЛЬКО колонки доски задач (в воронке продаж его нет).

## Модель данных (delta, аддитивно)

### Внутренние проекты
- `projects.type` text ('client' | 'internal'), NOT NULL DEFAULT 'client' + CHECK.
  Явный тип, не «угадывать по пустому company_id».
- Nullable сделать **три** колонки (для type='internal'):
  - `stage_id` → nullable (sync-триггер 019 уже переживает NULL);
  - `pipeline_id` → nullable (v1 пропустил; NOT NULL в живой БД);
  - `direction` → nullable (direction — атрибут продажи; расширять enum значением
    'internal' — хуже: 'internal' не направление бизнеса, и все места со switch по
    direction получат третью ветку).
- `stage` (legacy): **убрать DEFAULT 'new_lead'** (или INSERT internal с явным NULL) —
  иначе зеркало молча запишет стадию проекту вне воронки. Синергия со
  SPRINT-stage-deprecation: там stage и так становится производным.
- `status` для internal: won/lost не имеют смысла. Решение P1 — internal использует
  подмножество `open`/`on_hold` + завершение через `status='won'` НЕ делаем;
  вместо этого CHECK: `type='internal' AND status IN ('open','on_hold','completed')`
  → добавить значение `'completed'` в CHECK статусов (аддитивно) и маппить его в UI
  как «Завершён». Won/lost-виджеты фильтруют по type='client'.
- CHECK-инвариант целостности:
  `(type='client' AND pipeline_id IS NOT NULL AND stage_id IS NOT NULL AND direction IS NOT NULL)
   OR (type='internal' AND pipeline_id IS NULL AND stage_id IS NULL)` —
  чтобы «полу-internal» состояния не существовали в принципе.
- **Гейты S27, UI стадий, probability, lost-handling — только для type='client'.**
  S29 на уровне БД отваливается сам (см. выше), но UI-точки (PipelineBoard, ProjectCard
  chevron, ProjectDetail стадии, StackedPipeline, фильтры `stage !== 'won'/'lost'` в
  dashboard-content/CallModal/ActivityDrawer) должны проверять type/NULL. РАЗВЕДКА
  спринта: grep всех чтений stage_id/pipeline_id/direction/stage.

### Кастомные колонки канбана (per-project)
- `project_columns` { id, org_id NOT NULL, project_id NOT NULL → projects ON DELETE CASCADE,
  name, position, category, wip_limit? }. Колонки — ДАННЫЕ, не enum. Повесить
  существующий `trg_set_org_id` (org_id уже в живой БД, S23 применён).
- `category` — нормализующий класс. **Решение v2: в P1 категории делаем БИЕКТИВНЫМИ
  к lane**: `backlog` | `started` | `paused` | `done` (↔ next / now / wait / done).
  - v1 предлагал backlog/started/done/cancelled — но `wait` никуда не ложился
    (это не бэклог и не done), а cancelled в P1 не нужен (задач-отмен нет в модели).
  - Биекция делает бэкфилл без потерь и dual-write тривиальным (см. write-семантику).
  - `cancelled` добавим позже аддитивно, если появится реальная потребность —
    category это text + CHECK, не Postgres enum, расширяется дёшево.
  Зачем category вообще: у проектов разные ИМЕНА колонок, но личный борд и метрики
  сводят их к единому виду. Паттерн Linear (custom statuses + status category).
- `tasks.column_id` uuid nullable FK → project_columns **ON DELETE SET NULL** (learnings).
- `sort_order` меняет разрез: был per-lane, становится per-column для проектных задач.
  Бэкфилл обязан пересчитать порядок внутри новых колонок, иначе коллизии сортировки.

### Write-семантика lane ↔ column_id (НОВОЕ в v2 — это была главная дырка v1)
Dual-read без правил записи даёт задачу «Готово» на доске проекта и «now» на личном борде.
Правило — единый source of truth per task:

- Задача **с** `project_id`: истина — `column_id`. `lane` — **деривативная** колонка,
  синхронизируется BEFORE INSERT/UPDATE триггером на tasks из category колонки
  (category→lane однозначен благодаря биекции). Хуки lane для таких задач не пишут.
- Задача **без** `project_id`: истина — `lane`, как сейчас. `column_id IS NULL`.
- Перенос проектной задачи на ЛИЧНОМ борде: UI мапит целевую lane → category → колонка
  проекта с этой category, **первая по position** (если у проекта две колонки
  category='started' — берём первую; это осознанная потеря точности триажа, норм).
- Итог: `lane` не выпиливается вообще и перестаёт быть проблемой — S29 `create_task`
  пишет lane как раньше, личный борд работает как раньше, глобальные метрики читают lane.

### Дефолтные колонки и авто-назначение — на уровне БД, не клиента (НОВОЕ в v2)
- **Сидирование**: AFTER INSERT триггер на projects создаёт дефолтные колонки
  (Бэклог/В работе/Ожидание/Готово с category backlog/started/paused/done).
  Клиентское «создал проект → создай колонки» неатомарно и не покрывает другие
  точки создания проектов: `convert_lead()` (лид → сделка) создаёт проекты в SQL,
  минуя фронт. Триггер покрывает всех писателей разом. Для internal — те же дефолты.
- **Авто-назначение колонки**: BEFORE INSERT триггер на tasks — «если `project_id`
  задан, а `column_id` нет → колонка проекта, соответствующая lane задачи (или backlog)».
  Это закрывает S29 `create_task` (он пишет lane + project_id и ничего не знает про
  колонки), TaskQuickAdd, и любой будущий код. `create_task` в 029 НЕ трогаем.
- **Бэкфилл существующих задач**: одной миграцией — сидировать колонки всем текущим
  проектам, затем `column_id` = колонка проекта по биекции lane→category,
  пересчитать sort_order внутри колонок.

### Удаление/редактирование колонок (НОВОЕ в v2)
- Rename — свободно (category при этом не меняется без явного действия).
- Смена category у колонки — каскадный пересчёт `lane` задач в ней (тем же триггером).
- Удаление НЕпустой колонки: паттерн Jira/Linear — обязательный выбор
  колонки-приёмника («перенести N задач в …») в confirm-диалоге. ON DELETE SET NULL
  остаётся страховкой на уровне БД (осиротевшая задача рендерится в backlog-колонке
  борда как fallback), но UI до этого не доводит.
- Нельзя удалить последнюю колонку категории `done` и последнюю `backlog`
  (минимальный инвариант борда).

### RLS для project_columns (НОВОЕ в v2)
Текущая модель ролевая (`user_role()`): admin/pm — всё; member — своё по
created_by/owner_id/assigned_to. Нюанс: колонки — атрибут проекта, и member с задачей
в «чужом» проекте обязан ЧИТАТЬ колонки этого проекта, иначе его задача не отрисуется.

```
project_columns   | SELECT                          | INSERT/UPDATE/DELETE
admin, pm         | org-wide                        | org-wide
member            | колонки проектов, где есть его  | только проекты, где он
                  | задачи ИЛИ он owner проекта     | owner_id/created_by
viewer            | org-wide read                   | ✗
```

SELECT для member: `EXISTS (задача в проекте, assigned_to/created_by = uid)
OR owner проекта`. Это шире, чем policy на projects — закладывать в промпт явно.

### Секции/фазы внутри проекта (Фаза 2, опц.)
Для длинных внедрений: `project_sections` — группировка задач по этапу проекта
(Обследование/Настройка/Тестирование/ОЭ). Ортогонально колонкам: секция = «какой этап
проекта», колонка = «статус задачи». Решим на Фазе 2.

## Два разреза, оба остаются

- **Проектный борд** (новое, ядро): открыл проект → доска задач с кастомными колонками.
  Управление работой по проекту. Истина — `column_id`.
- **Личный борд** (существующий `/tasks`): «мои задачи» поверх всех проектов. Благодаря
  деривативному `lane` работает БЕЗ рефактора группировки — это триаж, не исполнение.
- Не конкуренты — паттерн Linear «My Issues vs Project view».

## Решения по открытым вопросам v1

1. **Навигация**: один список `/projects` с фильтром/табом по типу. Отдельный раздел
   умножает страницы ради одного WHERE. (Won/lost-аналитика фильтрует type='client'.)
2. **Дефолтные колонки**: единый фикс-набор для client и internal
   (Бэклог/В работе/Ожидание/Готово). Разные дефолты = уже шаблоны, это P3, не тащить в P1.
3. **Задача без проекта**: остаётся на `lane`, `column_id IS NULL`. С деривативным lane
   (см. write-семантику) вопрос «выпиливать ли lane» снят — не выпиливать, он дешёвый.

## Фазирование (каждая фаза ценна сама)

| Фаза | Скоуп | Оценка | Риск |
|---|---|---|---|
| **P1 — Ось + кастомные колонки + internal-проекты** | project.type + CHECK-инвариант, nullable stage_id/pipeline_id/direction, drop DEFAULT у stage, status 'completed', UI-гейтинг pipeline-логики по type, project_columns (+org_id+RLS) + tasks.column_id, триггеры: сид колонок, авто-назначение, lane-деривация, бэкфилл, проектный борд (KanbanBoard → динамические колонки) | ~1 спринт, тяжёлый (миграция + триггеры + KanbanBoard рефактор) | Средне-высокий |
| **P2 — Глубина исполнения** | секции/фазы, подзадачи (parent_task_id), зависимости (blocks/blocked-by), возможно category 'cancelled' | ~1 спринт | Средний |
| **P3 — Повторяемость (домен-киллер)** | шаблоны проектов (playbook внедрения → авто-генерация колонок+секций+задач), вехи (milestones) | ~0.7 спринта | Низкий (аддитивно) |

P1 — фундамент. P3 (шаблоны маркировочного внедрения) — реальная экономия часов на
каждом клиенте, но требует P1/P2 под собой.

## Главные риски / чек-лист P1

1. **Не перегрузить `stage_id` под доску задач** — две таблицы, два понятия.
2. **Тройка NOT NULL** (`stage_id`, `pipeline_id`, `direction`) + DEFAULT у `stage` —
   все четыре пункта в одной миграции, с CHECK-инвариантом. Проверить каждую UI-точку,
   читающую эти поля (grep-список в РАЗВЕДКЕ; БД-уровень уже NULL-safe: 019 и S29 проверены).
3. **Write-семантика** — триггер lane-деривации обязателен в P1, не «потом»: без него
   dual-read расползается с первого дня.
4. **KanbanBoard рефактор** — drag-drop с фиксированных lane на динамические колонки;
   самый большой кусок фронта. LaneColumn/AccordionLane параметризуются колонкой.
5. **Бэкфилл + sort_order** — сид колонок всем проектам, маппинг lane→category,
   пересчёт порядка. Прогнать на копии до прода (Supabase branch).
6. Migration-safety: строго аддитивно; `lane` не трогаем; S29/S27/019 не редактируем —
   вся совместимость через новые триггеры.
