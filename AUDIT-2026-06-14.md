## 1. P0 — Off-by-one «сегодня» (UTC vs local)

**Симптом (подтверждён в браузере):** в календаре сегодня (14 июня, Вс) подсвечен день **13**. 

**Корень:** `new Date().toISOString().slice(0, 10)` конвертирует в **UTC**. Для MSK (UTC+3) ночью/ранним утром UTC-дата на сутки меньше локальной.

Доказательство:
```
Локально MSK: 2026-06-14 02:30  →  UTC ISO: 2026-06-13T23:30Z
toISOString().slice(0,10) = "2026-06-13"   ← баг (показывает 13-е)
Intl(Europe/Moscow)        = "2026-06-14"   ← корректно
```

**Масштаб — 10 вхождений в 8 файлах** (не только календарь):
- `components/calendar/CalendarView.tsx:42` — подсветка «сегодня»
- `components/calls/CallTracker.tsx:18` — счётчик звонков за сегодня
- `components/layout/EventReminder.tsx:69` — напоминания о событиях
- `components/layout/ActivityDrawer.tsx:120,125` — ключ localStorage «фокус дня» (`focus-YYYY-MM-DD`)
- `components/layout/ActivityDrawer.tsx:188,278` — «звонки на сегодня», статы
- `components/shared/SmartAlerts.tsx:29` — алерты дедлайнов
- `components/dashboard/DashboardHome.tsx:199` — KPI «задачи на сегодня»

Последствие: «фокус дня» в drawer сбрасывается не в полночь по локали; задачи/звонки «на сегодня» считаются по неверной дате; дедлайны подсвечиваются на день раньше.

**CSV-экспорт** (`ContactsTable.tsx:199` — имя файла с датой) этим не страдает по смыслу, но для единообразия лучше тоже перевести на локальный хелпер.

**Фикс:** единый хелпер локальной даты в `lib/utils/date-helpers.ts`:
```ts
export function localDateKey(d: Date = new Date()): string {
  const y = d.getFullYear();
  const m = String(d.getMonth() + 1).padStart(2, '0');
  const day = String(d.getDate()).padStart(2, '0');
  return `${y}-${m}-${day}`;
}
```
Заменить все `new Date().toISOString().slice(0,10)` на `localDateKey()`.

## 2. P0 — tsc падает: MeetingModal

`npx tsc --noEmit` → **6 ошибок**, все в `components/meetings/MeetingModal.tsx`:

:41,42 Property 'company_id'/'contact_id' does not exist on type 'Meeting'

Модалка обращается к `company_id`/`contact_id`, которых нет в типе `Meeting` (и в Zod-схеме встречи). При этом схема БД (schema.md) утверждает, что у `meetings` эти колонки есть — значит либо тип `database.ts` устарел относительно БД, либо колонок реально нет.

**Действие:** сверить фактическую схему `meetings` (через Supabase MCP `list_tables`), привести в соответствие: либо добавить колонки в `meetings` + регенерировать `database.ts` + расширить Zod-схему, либо убрать обращения из модалки. Билд должен проходить (`next.config` имеет `ignoreBuildErrors: false`, так что это блокер деплоя).

> Прим.: tsc ругается только на эти 6 — остальной код типобезопасен.

## 3. P1 — График «Проекты по фазам» пустой при валидных данных

**Симптом (подтверждён в браузере + DOM):** на `/analytics` график «Проекты по фазам» показывает оси (Привлечение/Проработка/Согласование/Закрытие) но **ноль баров**. При этом 3 открытых проекта реально распределены: negotiate=2 (ОМК, Завод Атлант), close=1 (Аграрная группа) — проверено прямым запросом к БД.

DOM-замер: контейнер 717×192 (нормальный размер), `<Bar>` групп смонтировано 2, но `<path>` баров — **0**.

**Данные и цвета корректны** (`Charts.tsx:164-171` маппит фазы верно; `getPhaseForStage('trilateral_meeting')→negotiate`, `('contract_review')→close`; градиенты `AURA_PHASE` совпадают с ключами фаз). Значит баг — в рендере Recharts.

**Вероятная причина:** `isAnimationActive={isAura}` (`Charts.tsx:201`) — в Aura бары анимируются с нулевой ширины; при `layout="vertical"` и данных, приходящих после маунта, Recharts оставляет бары схлопнутыми в 0. Тот же `isAnimationActive={isAura}` в `CallsChart.tsx`.

**Фикс (в порядке вероятности):**
1. Убрать `isAnimationActive` или сделать `isAnimationActive={false}` для vertical bar (самый надёжный для data-driven).
2. Если анимацию надо сохранить — добавить `key` на `<BarChart>` от длины данных, чтобы форсить ремаунт после прихода `projects`.
3. Проверить, что `ResponsiveContainer minHeight={0}` не схлопывает на первый кадр — задать явный `minHeight`.

> График «Звонки по неделям» **пустой корректно**: ось начинается с 27 апр, а последние звонки — неделя 6 апр (проверено). Это не баг, но стоит добавить empty-state «нет данных за период».

## 4. P1 — RLS: `meeting_attendees` открыт всем

Advisor + прямой запрос политик:
```
meeting_attendees: policy "attendees_all" FOR ALL USING (true)  ← без фильтра по пользователю
```
Любой аутентифицированный (а по advisor — и `anon`) может читать/писать все строки `meeting_attendees`. Таблица в коде сейчас **не используется** (нет `.from('meeting_attendees')`), но дыра реальна — если завтра туда польются участники встреч с ФИО/контактами, это утечка PII (152-ФЗ).

**Фикс:** заменить на скоуп через связь с `meetings`:
```sql
DROP POLICY attendees_all ON meeting_attendees;
CREATE POLICY attendees_own ON meeting_attendees FOR ALL
  USING (EXISTS (SELECT 1 FROM meetings m WHERE m.id = meeting_attendees.meeting_id AND m.user_id = (select auth.uid())))
  WITH CHECK (EXISTS (SELECT 1 FROM meetings m WHERE m.id = meeting_attendees.meeting_id AND m.user_id = (select auth.uid())));
```

Похожий, но менее острый кейс — `activity_log` политика `Service insert logs` с `WITH CHECK(true)`: нужна для SECURITY DEFINER триггеров, но в текущем виде позволяет клиенту вставлять произвольные лог-записи. Низкий риск (лог), но стоит ограничить ролью или убрать (триггеры пишут как DEFINER и так).

**Фикс:** обернуть в подзапрос — `auth.uid()` → `(select auth.uid())`. Затронуты политики на: activities, activity_log, call_tracker_days, calls, companies, contacts, dashboard_sync, kpi_entries, leads, meetings, profiles, project_files, projects, scheduled_calls, tasks, user_settings (полный список политик — в приложении).

## 9. P3 — Supabase security гигиена

- **14 функций** с mutable `search_path` (`update_updated_at`, все `log_delete_*`, `handle_new_user`, `convert_lead`, `user_role`, `log_stage_change`, `sync_deal_stage_fields` и др.) — добавить `SET search_path = public, pg_temp`. [[линтер 0011](https://supabase.com/docs/guides/database/database-linter?lint=0011_function_search_path_mutable)]
- **14 SECURITY DEFINER функций** доступны роли `anon` через `/rest/v1/rpc/*` (включая `convert_lead`, `log_delete_*`). Триггер-функции не должны быть вызываемы напрямую — `REVOKE EXECUTE ... FROM anon, authenticated`. [[линтер 0028/0029](https://supabase.com/docs/guides/database/database-linter?lint=0028_anon_security_definer_function_executable)]
- **Leaked password protection выключена** — включить HaveIBeenPwned в Auth settings. [[доки](https://supabase.com/docs/guides/auth/password-security)]
- **multiple_permissive_policies**: `activity_log`/INSERT и `profiles`/SELECT имеют по 2 пересекающиеся политики — объединить.

## 10. P3 — Производительность БД и дрейф схемы

- **24 FK без индекса** (`*_created_by_fkey`, `*_contact_id_fkey`, `meeting_attendees_*`, `scheduled_calls_*`, `projects_stage_id_fkey` и др.) — JOIN'ы и каскады деградируют.
- **28 неиспользуемых индексов** (`idx_projects_*`, `idx_meetings_*`, `idx_tasks_*`, `idx_activities_*` и др.) — кандидаты на DROP. Парадокс: индексировали `owner/status/date`, а FK на `created_by/contact_id` остались без индексов. При фиксе FK-индексов часть unused можно дропнуть в той же миграции.
- **Дубли `stage`/`stage_id`** в `projects` (и `loss_reason`/`lost_reason`, `stage`/`status`) — признак незавершённой миграции Sprint-1. Оба набора живут одновременно. Риск рассинхрона: аналитика читает `stage`, канбан — `stage_id`. Решить, какой источник истины, второй депрекейтить.
