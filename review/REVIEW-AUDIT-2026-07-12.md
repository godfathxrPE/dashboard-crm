## Согласие по частям

### Часть 1. Код — P1 (безопасность и потеря данных)

**Полностью согласен.** Это правильный топ приоритетов:

1. **1.1 token.json** — критичный инцидент; даже в .gitignore файл физически уезжает с копиями (как эта папка на аудит).

2. **1.2 миграции ≠ прод** — самая опасная системная проблема. Репозиторий перестал быть source of truth: `attendees_all`, `Service insert logs WITH CHECK(true)`, отсутствие `pipelines` в цепочке — всё подтверждено. Без ревизионного снимка любой staging/DR — лотерея.

3. **1.3 + 1.4 + 1П-4** — связка «тихая потеря ввода»:
   - клик мимо формы без guard;
   - ошибка сети/RLS без UI-фидбека;
   - длинная модалка сделки не влезает в 784px.
   
   Для CRM это дороже любого визуального бага. Решение аудита (общий Modal-wrapper + sonner + mutation defaults) — архитектурно верное.

4. **1.5 realtime** — баг реальный и коварный: shared channel name + `removeChannel` при unmount любого подписчика. Параметр `_queryKey` действительно мёртвый.

5. **1.6 ExcelImport** — подтверждён; partial import без rollback — классический data-corruption сценарий.

### Часть 1. Код — P2

**Согласен с приоритетом, с нюансами:**

- **2.3 ai_hub org_id** — реальная дыра, но exploitability ниже, чем у `attendees_all`: нужен доступ к чужому `entity_id` через RLS calls/meetings. Всё равно чинить — это единственный tenant-INSERT без org-границы.

- **2.4 инвайты без token** — валидно при выключенной верификации email. Severity зависит от настроек Supabase Auth; аудит правильно помечает как архитектурный риск.

- **2.5 legacy stage** — технический долг с июня, подтверждён комментарием в коде. «Путь B» действительно не сделан.

- **2.7 full-table fetch** — архитектурно верно для текущего data-слоя (`select('*')` + layout-хуки). На 5–10k записей станет ощутимо; не блокер сегодня, но в дорожной карте на месте.

- **2.8 hydration** — подтверждён; inline-script в root layout чинит CSS-класс, но не дерево React — точная формулировка.

### Часть 1. Код — P3

**В основном согласен:**

- **3.1 мёртвый код** — направление верное, но уточнение:
  - `ui/Watermark` (старый) — мёртв; жив `WatermarkNew` (7 импортов)
  - `widgets/*` — barrel не импортируется, но `TasksSidebar` используется в `KanbanBoard`; `ActivityDrawer` держит **локальную** копию `StatsWidget`
  - Счёт «16 файлов» завышен на ~3–4, суть (шум в навигации) верна

- **3.3 database.ts** — критично для долгосрочной поддержки; placeholder с ручными типами — факт.

- **3.8 дубль 005** — подтверждён, риск случайного apply реален.

## Замечания и расхождения

### 1. Два аудита одного дня — риск двойной работы

Visual-audit (темы/контраст) и этот комплексный аудит пересекаются в:
- тёмных кнопках, yellow-токенах, kanban headers, микрокегле.

**Рекомендация:** при планировании спринтов явно пометить «уже закрыто visual-audit P0/P1» и не дублировать. Этот файл — source of truth для **кодовых** и **продуктовых** проблем; visual-audit-report — для токен-матрицы.

### 2. Сводный порядок работ — согласен, одно уточнение

Пункт «жёлтый контраст — 15 минут» (1П-1) — реалистично **только для CSS-override**. Полный fix модалок (1.3 + 1П-4 + toast) — это отдельный спринт ~2–3 дня, не «сегодня после 15 минут».

Предлагаемая последовательность:
1. **Сегодня:** revoke сессий (если token.json разъезжался) + 1П-1 yellow CSS
2. **Спринт 1:** toast + Modal-wrapper (isDirty + viewport) + realtime fix
3. **Спринт 2:** миграция-снимок + RLS бэкпорт + ai_hub/invites org
4. **Спринт 3:** дефолт aura → удаление scandi → общий shell

### 3. 2.6 DELETE projects — severity

Верно описано, но frequency низкая (редкое удаление проекта с историей). Не P1, согласен с P2.

### 4. 3.7 CORS `*` в edge functions

Риск завышен при наличии JWT-проверки; это P3/gигиена, не P1. Согласен с классификацией аудита.

### 5. «tsc чистый» — перепроверить перед спринтом

Аудит утверждает чистый tsc. На момент ревью не гонялся повторно — стоит верифицировать в CI перед «доверие к вводу», т.к. MeetingModal ломался в июньском аудите.
