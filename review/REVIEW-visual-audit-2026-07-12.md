## Замечания и расхождения

### 1. Definition of Done слишком строгий

DoD спринта: «0 FAIL текстовых пар» на всех темах. Из 84 FAIL значительная доля — **`kind: "ui"`** (границы `border/border2` на surface, ratio 1.1–2.8:1). Это **не текст**, и для декоративных разделителей пониженный контраст допустим по WCAG 1.4.11, если граница не единственный индикатор.

**Рекомендация:** разделить DoD:
- **Текстовые пары** (`kind: text`, включая бейджи и кнопки): 0 FAIL.
- **UI-пары** (декоративные border): отдельный KPI; для инпутов — `--border-input` ≥3:1 (как в P2 спринта).

Иначе команда будет «гоняться» за 1.29:1 у aura border на white, хотя это не блокер для текста.

### 2. Scandi-dark: white-on-white в скрипте — частичный false positive

Скрипт показывает `white / bg-accent = 1.0:1` для scandi-dark, потому что `--accent: #ffffff`. В реальном UI Scandi **переопределяет** primary-кнопки в outline-стиль (`.t-scandi button.bg-accent` — прозрачный фон, `color: var(--text)`). Live-кнопки могут быть читаемы, хотя токен-матрица «красная».

**Рекомендация:** в отчёте/скрипте пометить Scandi как «theme overrides button fill» или добавить в скрипт флаг `theme_button_override: true` для scandi, чтобы не раздувать счётчик FAIL.

### 3. Opacity — найден StackedPipeline, но не весь класс проблем

Аудит правильно фокусируется на StackedPipeline, но по кодовой базе `opacity-50` на **информативном** контенте встречается ещё:

- `DashboardHome.tsx` — иконки/счётчики пустых виджетов: `text-text-mute opacity-50`
- `.t-scandi .pill-*` — `opacity: 0.6` на статусах (компенсируется формой ● ○ −)
- Массовый `disabled:opacity-50` на кнопках — для disabled **приемлемо** (не информативный контент в активном состоянии)

**Рекомендация:** после P0-4 прогнать grep `opacity-(40|50|60)` × `text-` и зафиксировать правило в `theme-system.md` (спринт это предлагает — хорошо).

### 4. Несогласованность путей артефактов

- `review/contrast_results.json` — вход для HTML-отчёта
- Скрипт пишет в `scripts/audit-contrast-results.json`

Два файла с одинаковым содержимым в разных папках — риск рассинхрона. **Рекомендация:** один canonical path + symlink или аргумент `--out`.

### 5. Frost недооценён в спринте

По матрице frost — **13 FAIL** (хуже aurora/tidal). В волне 1 фиксы frost/aurora/tidal сгруппированы одинаково, но frost дополнительно страдает от purple badge (4.0:1) и accent badge (4.0:1). После волны 1 стоит **отдельно перепроверить frost** — возможно, понадобятся чуть более тёмные `*-text` для badge-пар.

### 6. Sand `white / bg-accent = 4.5:1` — пограничный PASS/FAIL

В JSON `pass: false` при ratio ровно 4.5 — скорее floating point (4.496 → round 4.5). Не баг аудита, но при фиксе sand accent-button стоит целиться в **≥4.6:1** с запасом.

### 7. Мелочь в спринте: `--font-app` для Fuji

Спринт предлагает `.t-fuji { --font-app: var(--font-plex) }`, но сейчас Fuji задаёт `font-family` напрямую на `.t-fuji` (строка 506). При миграции на `--font-app` **удалить** дублирующее правило, иначе специфичность `.t-fuji` может снова перебить `html`.

## Оценка скрипта `audit-contrast.py`

**Сильные стороны:**
- Парсит реальный `globals.css`, каскад `:root` → тема → scandi-dark media query
- Резолвит `var()` с fallback, композитит rgba над bg
- Двойная метрика WCAG + APCA, OKLCH для hue-анализа
- Явное предупреждение про opacity на контейнерах

**Что добавить (не блокирует спринт):**
- Параметр `--opacity 0.5` для симуляции StackedPipeline
- Отдельная категория `kind: button` с учётом theme-specific overrides (scandi, aura)
- Exit code 1 при N text-FAIL (для CI)
- Единый output path

Пересчёт APCA на канонических значениях заявлен в комментарии — доверяю; полная валидация SAPC выходит за рамки этого ревью.
