P0 — системные проблемы

P0ТИПОГРАФИКАШрифтовая система тем не работает: всё рендерится Manrope

Темы задают фирменные шрифты на html (.t-aura → Onest, .t-scandi → Inter), но правило body { font-family: var(--font-manrope) } (globals.css:1346–1347) перекрывает наследование для всего контента. Собственная декларация элемента всегда сильнее унаследованной.

Live-проверка (t-aura, /deals): html → Onest, Manrope, system-ui ← замысел body → Manrope, system-ui ← реальность (весь UI) h1 → Unbounded (работает: правило бьёт в h1 напрямую) Затронуты: Aura (Onest не виден нигде кроме заголовков), Scandi (Inter не виден). Fuji работает случайно — там есть отдельное правило .t-fuji body.

Фикс: убрать font-family из body (оставить background/color) или дублировать тем-селекторы на body: .t-aura body { font-family: … }. Загруженные, но неиспользуемые Onest/Inter — это ещё и ~100–200KB лишних webfonts в каждой загрузке.

P0ЦВЕТУтечка --yellow-text: #92661d из :root во все тёмные темы

:root определяет светлотемный токен --yellow-text. Класс .text-yellow { color: var(--yellow-text, var(--yellow)) } задуман с fallback, но fallback не срабатывает: тёмные темы не переопределяют --yellow-text, и тёмно-янтарный текст для светлого фона попадает на тёмный фон.

COMPUTED — жёлтый текст (предупреждения, "Бюджет", дедлайны): frost: #92661d на surface = 3.16:1 FAIL (нужно 4.5) aurora: #92661d на surface = 3.38:1 FAIL tidal: #92661d на surface = 3.59:1 FAIL scandi-dark: #92661d на bg = 3.72:1 FAIL бейджи (bg-yellow-l): 2.30–3.41:1 — хуже всего

Фикс: перенести --yellow-text из :root внутрь светлых тем, а в тёмных задать светлые варианты (например frost: #F0C45E — его же --yellow уже даёт ≥7:1 на тёмном). Побочный эффект того же бага: янтарь просачивается в монохромный Scandi (см. P2-консистентность).

P0ЦВЕТТёмные темы: белый текст на solid-кнопках 1.6–3.9:1

Button primary и CTA — bg-accent text-white. В тёмных темах акценты светлые, и белый текст проваливается. Aura уже решила эту задачу (тёмные *-text заливки под белый текст), Scandi — переопределением кнопки в outline. Frost/Aurora/Tidal — не решили (зафиксировано и в HANDOFF).

COMPUTED — белый на заливке: frost: accent #5b8aff = 3.22 · green #5bb88e = 2.42 · blue #5fa8e8 = 2.54 · red #e8795a = 2.87 aurora: accent #a060ff = 3.74 · green #20e8a0 = 1.60 · blue #40b8ff = 2.21 · red #ff6090 = 2.87 tidal: accent #48b890 = 2.46 · green #40c088 = 2.30 · blue #5ca8d0 = 2.64 · red #c86858 = 3.77

Фикс (перенести приём Aura): тёмный текст на ярких заливках — .t-frost .bg-accent { color: #0d1020 } даёт 6.5:1 с тем же брендовым цветом (яркие кнопки на тёмном фоне выглядят органично, паттерн Vercel/Linear dark). Альтернатива — затемнить заливки (#4a70cf / #8e55e3 / #328165), но это меняет характер тем.

P0ЦВЕТ · LIVEStackedPipeline: opacity-50 убивает текст будущих этапов — 1.69–1.99:1

Деталь проекта, дорожки delivery-пайплайна: future-треки получают opacity-50 на контейнере (StackedPipeline.tsx:127) — любой токен под ним теряет половину контраста. Худшие значения во всём приложении. Стандартный контраст-аудит этого не видит: расчёт по токенам давал PASS, провал нашёлся только opacity-aware замером живого DOM.

Live, деталь проекта (COMPUTED, с учётом эффективной opacity): «Установка БИТ.MDT», «Запуск», «Передача на поддержку»… text-mute × op0.5 → #B0B0BE на #F6F6FA = 1.99:1 (11px) FAIL Лейблы фаз «Планируется» / «Исполняется» / «Завершён» track-цвет × op0.5 → 1.69–1.93:1 FAIL

P2 — гигиена и консистентность

P2ТИПОГРАФИКАМикрокегль как норма: 206 мест с 9–11px

По кодовой базе (COMPUTED, grep): text-[9px] ×13 · text-[10px] ×146 · text-[11px] ×47 text-xs (12px) ×413 · text-sm (14px) ×238 · text-base (16px) ×2

Рабочий кегль CRM — 12–13px, это осознанная плотность и для десктопного CRM приемлемо. Но 9–10px в meta-строках (даты, счётчики, лейблы вроде «Лид» на 9px) в сочетании с text-mute — двойной штраф: мелко И бледно. WCAG формально не нормирует кегль, но AA-контраст рассчитан на «normal text», а не на 9px.

Фикс-политика: минимум 11px для любых информативных надписей; 9px запретить; 10px — только для неинформативных декоративных меток. Для 10–11px использовать text-dim, не text-mute.

P2UIГраницы инпутов ниже 3:1 во всех темах

COMPUTED, border на surface: aura 1.29 · washi 1.47 · frost 1.17 · fuji 1.42 · scandi 1.60 · tidal 1.13

Для декоративных разделителей это норм. Но Input использует border-border как единственную границу поля — WCAG 1.4.11 требует 3:1 для визуальных границ контролов, если они единственный индикатор. Смягчает: у инпутов есть фон surface на bg и placeholder.

Фикс: для форм-контролов использовать border2 и поднять его до ≥3:1 против surface (aura: #A6A6BC ≈ 3.0) — только в контексте инпутов, не глобально.

Видно на живых скриншотах. Реши на уровне концепции: либо это фича (единственный цветовой сигнал = warning), либо баг. Сейчас это выглядит случайностью — часть статусов монохромные, часть янтарные.

Методология

1) Парсинг переменных всех 9 тем из globals.css (+ Scandi dark из media query), разрешение var()-цепочек и fallback'ов, композитинг rgba поверх фоновых слоёв. 2) WCAG 2.2 relative luminance по формуле спецификации; APCA-W3 0.0.98G-4g; обе реализации сверены с каноническими значениями (#767676/#fff = 4.54, APCA #888/#fff = 63.06). 3) Live-аудит: обход DOM на живом сайте (t-aura и t-scandi), вычисление эффективного фона по цепочке предков. 4) CVD: симуляция Machado et al. 2009 (severity 1.0), ΔE Lab. 5) Типографика: полнотекстовый grep по кодовой базе на устройстве. 6) Live-проход 13 разделов Aura с opacity-aware расчётом (эффективный цвет текста с учётом каскадной CSS opacity).

Ограничения: пары с градиентными подложками (орбы Aura) считались по базовому фону — реальный контраст локально может отличаться на ±0.3; тёмный Scandi проверен по коду (media query), не live; hover/focus-состояния оценены по токенам, не по всем комбинациям. Прод-сборка на Netlify отстаёт от локального working tree (разные версии globals.css) — код-аудит сделан по локальной версии, live-аудит по деплою; ключевые выводы совпадают в обеих.
